请基于当前硫化排程项目，实现滚动排程衔接逻辑，要求最小改动，不重写现有排程引擎。

业务背景：
硫化排程采用滚动窗口模式，一次排 1 天 + 预排 2 天。例如：
第一次排程窗口：2026-04-23 ~ 2026-04-25
第二次排程窗口：2026-04-24 ~ 2026-04-26
每个窗口包含 8 个班次。
第二次排程需要承接第一次排程结果，避免重复排程、重复扣减月计划 SKU、错误判断机台状态、续作、换模、换活字块。

总体要求：
在现有排程主流程前增加一个 RollingScheduleBridge / RollingPlanCarryOverService，不大改原有排程引擎。该服务负责读取上一版排程结果，识别已滚出区、重叠区、新增区，并生成本次排程的初始状态。

窗口规则：
1. prevWindow 表示上一版排程窗口。
2. currWindow 表示本次排程窗口。
3. rolledOutTasks = 上一版中早于本次窗口开始时间的任务，例如 2026-04-23。
4. overlapTasks = 上一版和本次窗口重叠的任务，例如 2026-04-24 ~ 2026-04-25。
5. newWindow = 本次窗口中上一版不存在的部分，例如 2026-04-26。
6. 判断窗口时以班次开始时间和结束时间为准，不要只按自然日判断。

冻结策略：
第一版按最小改动实现：
1. 已滚出区任务不再重排，只用于扣减计划量和推进机台状态。
2. 本次窗口第 1 天的重叠任务默认冻结继承，例如 2026-04-24。
3. 本次窗口第 2 天的重叠任务可以先默认冻结继承，后续再通过参数支持可重排，例如 2026-04-25。
4. 新增窗口正常调用原有排程逻辑生成，例如 2026-04-26。
5. 冻结继承任务需要作为 lockedTasks 放入机台日历，后续新任务不能与其时间冲突。

计划量衔接：
1. 第二次排程前，必须先扣减上一版已经承诺的计划量。
2. 对每个 monthPlanSku 计算：
   availableQtyForCurrentRun =
       originalRemainingQty
       - rolledOutQty
       - lockedCarryOverQty
       - mesActualProducedQty
       - releasedQty
3. 如果 availableQtyForCurrentRun < 0，则按 0 处理，并记录异常日志。
4. 排程引擎只能消费 availableQtyForCurrentRun > 0 的 SKU。
5. 不建议直接修改数据库中的原始月计划量，优先在 DTO 中增加临时字段：
   originalRemainingQty
   committedQty
   rolledOutQty
   lockedQty
   actualProducedQty
   availableQtyForCurrentRun

月计划 SKU：
1. 建立 Map<monthPlanSkuKey, occupiedQty>。
2. key 不要只使用 materialCode，建议包含 monthPlanId、materialCode、spec、pattern、bomVersion 等项目中已有可用字段。
3. 防止第二次排程重复消费第一次已经冻结或已滚出的 SKU 数量。
4. 如果继承任务找不到对应月计划 SKU，需要保留任务，但记录 CARRIED_TASK_MONTH_PLAN_NOT_FOUND 异常。

机台状态衔接：
1. 第二次排程的机台初始状态不能只使用 MES 当前状态。
2. 需要按时间顺序推进：
   MES 当前在机状态
   -> 上一版滚出区最后任务
   -> 本次冻结继承区最后任务
3. 对每台机生成 MachineState：
   machineCode
   availableTime
   lastMaterialCode
   lastSpec
   lastPattern
   lastEmbryoDesc
   lastMainPattern
   lastMoldCode
   lastTypeBlockCode
   lastTaskId
   lastStateSource
4. machineAvailableTime =
   max(
       本次窗口开始时间,
       MES 当前在机预计结束时间,
       上次滚出区该机最后任务结束时间,
       本次冻结区该机最后任务结束时间
   )

续作衔接：
1. 续作判断基于承接后的 MachineState。
2. 续作来源优先级：
   MES 当前在机
   > 上次已下发排程最后任务
   > 上次排程滚出区最后任务
   > 本次冻结继承任务最后任务
3. 注意不要在续作识别阶段立即消费月计划 SKU。
4. 先识别所有机台的续作候选关系，再根据 SKU 可用量统一分配，避免同一 materialCode 同时出现在多台机时只有第一台机获得续作资格。

换模衔接：
1. 换模判断基于承接后的 MachineState.lastMoldCode。
2. 如果 candidateSku.requiredMoldCode == lastMoldCode，则不换模。
3. 如果不一致，则按原有逻辑生成换模任务。
4. 如果上一次排程中冻结区已经包含换模任务，本次必须继承该换模任务，并锁定其时间段。
5. 继承换模任务后，需要更新机台状态为换模后的模具、物料、规格、花纹等信息。

换活字块衔接：
1. 换活字块判断基于 MachineState.lastTypeBlockCode 或项目中已有的胎侧文字、品牌、规格、花纹、胎胚描述等字段。
2. 如果 requiredTypeBlockCode == lastTypeBlockCode，则不换活字块。
3. 如果不一致，则按原有逻辑生成换活字块任务。
4. 对续作收尾机台，候选 SKU 排序优先级增加：
   a. 同胎胚描述 + 同主花纹
   b. 同规格
   c. 同活字块
   d. 换活字块耗时短
   e. 计划紧急度高
5. 第一版可只在原有排序上增加 sameEmbryoDescAndMainPattern 和 sameSpec 两个排序项。

任务来源标识：
请为本次排程结果中的任务增加或复用以下标识：
taskSource：
1. ROLLED_OUT_REFERENCE：已滚出引用，只用于扣量和状态推进
2. CARRIED_OVER_LOCKED：从上一版继承并冻结
3. REPLANNED：重排生成
4. NEW：本次新增生成
5. MES_ACTUAL：MES 实绩

如果已有类似字段，请复用，不要重复造字段。必要时可以使用扩展字段保存。

异常处理：
1. 上一版排程不存在：走现有首次排程逻辑。
2. 上一版与本次窗口无重叠：上一版只用于扣减已执行/已下发量，不做冻结继承。
3. MES 实绩与上一版排程不一致：优先使用 MES 实绩，记录 PREVIOUS_PLAN_MES_MISMATCH。
4. 继承任务找不到月计划 SKU：保留任务，记录 CARRIED_TASK_MONTH_PLAN_NOT_FOUND。
5. 扣减后月计划量为负：availableQtyForCurrentRun = 0，记录 MONTH_PLAN_OVER_COMMITTED。



最终推荐落地方式
P0 必做：
1. 上一版排程读取
2. 窗口重叠识别
3. 已滚出区扣量
4. 重叠区冻结继承
5. 机台 lockedTasks
6. 机台 lastState 承接

P1 建议做：
1. 月计划 SKU 占用明细
2. 续作候选与 SKU 消费解耦
3. MES 实绩优先
4. 换模、换活字块状态承接

P2 后续优化：
1. 4/25 可配置重排
2. 冻结天数参数化
3. 异常可视化
4. 多版本排程差异对比


一句话总结：
第二次排程不是重新排 4/24 ~ 4/26，而是先继承并锁定第一次排程中仍然有效的 4/24 ~ 4/25，扣减 4/23 和冻结区已经占用的计划量，再基于承接后的机台尾状态去排新增的 4/26。

实现要求：
1. 尽量不要改原有排程引擎核心算法。
2. 优先新增 RollingScheduleBridge / RollingPlanCarryOverService。
3. 原有排程引擎只需要接收：
   adjustedMonthPlanSkus
   machineInitialStates
   lockedTasks
   machineCalendar
   continuationCandidates
4. 保持原有续作、换模、换活字块逻辑可复用。
5. 增加必要单元测试和集成测试。

测试场景：
1. 第一次排 2026-04-23 ~ 2026-04-25，第二次排 2026-04-24 ~ 2026-04-26，验证 4/24、4/25 不重复排。
2. 验证 4/23 已滚出任务会扣减月计划 SKU。
3. 验证冻结继承任务会锁定机台时间。
4. 验证 4/26 会基于 4/25 继承后的机台状态继续排。
5. 验证同一 materialCode 在多台机续作时，不会只有第一台机消费月计划 SKU。
6. 验证换模判断使用承接后的 lastMoldCode。
7. 验证换活字块判断使用承接后的 lastTypeBlockCode。
8. 验证 MES 实绩与上一版排程不一致时，优先使用 MES。
9. 验证扣减后月计划量为负时不会继续排该 SKU，并记录异常。
