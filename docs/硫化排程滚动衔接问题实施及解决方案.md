## 三、问题发现

### 问题1：第二次排程中 3302002531 的 IS_END 从1变为0

**现象**：
- 第一次排程：3302002531 的 IS_END = 1（已收尾）
- 第二次排程：3302002531 的 IS_END = 0（未收尾）

**根因分析**：
3302002531 的月计划余量（MOULD_SURPLUS_QTY）在两次排程中均为30，但其 DAILY_PLAN_QTY 第一次为30，第二次为22。

第一次排程中：
- 3302002531 班次2=22 + 班次3=8 = 30 = surplusQty(30) → IS_END=1

第二次排程中：
- 3302002531 仅班次8=22 < surplusQty(30) → IS_END=0
- 这是因为第二次排程窗口只有1天（4/26），3302002531只排了1个班次

**影响**：3302002531 在第二次排程中未被标记为收尾，可能导致后续排程继续为其分配机台，造成产能浪费。

**解决方案**：
收尾判定逻辑需要综合考虑：如果 SKU 的月计划余量 ≤ 本次排程的总排产计划量，仍应标记为收尾。当前 `isEnd` 的判定是在每个排程结果维度做的（`finalPlanQty >= surplusQty`），但 3302002531 在第二次排程的总排产量为22，小于 surplusQty=30，所以未标记收尾。这是**正确的业务逻辑**——22 < 30 确实没有收尾。但需要确认：第二次排程的 surplusQty 是否应该更小？

**关键疑问**：第二次排程中 3302002531 的 surplusQty=30 是否准确？
- 月计划 TOTAL_QTY = 140
- 日完成量：4/20=44, 4/21=66，月累计=110
- 截至T-1日(4/23)的累计完成量：如果没有4/22和4/23的完成量数据，则累计仍为110
- surplusQty = 140 - 110 = 30 → **看起来正确**

但第一次排程已经为 3302002531 安排了 30 的计划量（班次2=22+班次3=8），这部分计划量在第二次排程中是否应该从余量中扣减？

**问题本质**：当前余量计算仅扣减"已完成量"，未扣减"前次排程已安排但尚未完成的计划量"。这导致滚动排程时余量偏大。

### 问题2：第二次排程班次分布不合理——3302001884 排了班次2-5，3302002531 仅排了班次8

**现象**：
- 3302001884：班次2=22, 班次3=22, 班次4=22, 班次5=22（共88）
- 3302002531：班次8=22（共22）

两次排程都只用了 K2024 一台机台，但第二次排程中 3302002531 被排到了班次8（最后一个班次），而 3302001884 占据了班次2-5。

**根因**：第二次排程中，3302002531 的 pendingQty 可能很小（窗口内无日计划量，且继承量已扣减），因此被排在剩余产能的末端。

**影响**：3302002531 生产延后，可能导致交期延误。

---

### 问题3：两次排程班次分布缺乏滚动衔接的连续性

**现象**：
- 第一次排程（窗口[4/23, 4/25]）：3302002531在班次2-3，3302001884在班次5-8
- 第二次排程（窗口[4/24, 4/26]）：3302001884在班次2-5，3302002531在班次8

**期望**：第二次排程应该继承第一次排程的重叠窗口班次，3302002531在班次2-3的安排应该被继承到第二次排程中。

**实际情况**：查看 RollingScheduleHandoffService 代码，第二次排程确实有滚动衔接逻辑。但从结果看，3302002531 在第二次排程的班次8=22，这应该是**新增排产部分**，而非继承部分。继承部分（班次2-3）的数据可能已被合并到 3302001884 的班次2-5中。

**验证方式**：需要检查第二次排程的 `rollingInheritedScheduleResultList`，确认继承部分是否正确。

### 问题4：3302001884 在未排产表中同时存在排程结果

**现象**：
- 排程结果表：3302001884 有排程结果（88条/22条）
- 未排产表：3302001884 也有记录（第一次396条，第二次88条）

**根因**：3302001884 在排程过程中先被添加到排程结果中，但由于其他机台分配失败，剩余的待排产量被写入未排产表。这表明只有 K2024 一台机台能生产 3302001884，其余机台都无法匹配。

**影响**：大量的未排产需求（396/88），说明机台配置严重不足。

### 问题5：TOTAL_DAILY_PLAN_QTY 两次排程均为396/140，未体现滚动差异

**现象**：
- 3302001884：TOTAL_DAILY_PLAN_QTY=396（=月计划TOTAL_QTY）
- 3302002531：TOTAL_DAILY_PLAN_QTY=140（=月计划TOTAL_QTY）

**问题**：TOTAL_DAILY_PLAN_QTY 应该取月计划 TOTAL_QTY，目前取值正确。但字段名"DAILY"可能引起歧义——它实际是月计划总量，而非日计划量。

### 问题6：DAILY_PLAN_QTY 取值不一致

**现象**：
- 第一次排程 3302001884：DAILY_PLAN_QTY=88
- 第二次排程 3302001884：DAILY_PLAN_QTY=88
- 第一次排程 3302002531：DAILY_PLAN_QTY=30
- 第二次排程 3302002531：DAILY_PLAN_QTY=22

**根因**：DAILY_PLAN_QTY 取自 `plan.getDayVulcanizationQty()`=33，但排程结果中显示为88/30/22，说明该值在排程过程中被重新计算过（可能是班次计划量汇总）。

3302001884：88 = 4班×22（第一次4个班次，第二次4个班次）
3302002531：30 = 22+8（第一次2个班次），22 = 22（第二次1个班次）

**问题**：DAILY_PLAN_QTY 实际反映的是"本次排程安排的班次计划量总和"，而非月计划中的"日硫化量"。这与其字段名"日计划量"的含义不完全一致。如果业务期望 DAILY_PLAN_QTY 取8班班次计划量总和（即33×8/3≈88），则需要修改取值逻辑。

### 问题7：模具交替计划存在时间冲突

**现象**：
| 排程日期 | 前物料 | 后物料 | 换模时间 |
|---------|--------|--------|---------|
| 04-25 | 3302001556 | 3302002531 | 2026-04-23 06:00 |
| 04-25 | 3302002531 | 3302001884 | 2026-04-24 06:00 |
| 04-26 | 3302002531 | 3302001884 | 2026-04-24 06:00 |
| 04-26 | 3302001884 | 3302002531 | 2026-04-26 06:00 |

**问题**：
1. 第一次排程中 3302001556→3302002531 的换模在 4/23 06:00，但第二次排程中这个换模不存在——因为第二次的排程窗口从4/24开始，4/23的部分由继承逻辑处理
2. 两次排程在 4/24 06:00 都有 3302002531→3302001884 的换模，这是正确的（属于重叠窗口）
3. 第二次排程在 4/26 06:00 有 3302001884→3302002531 的换模（回切），这是新增排产部分

**潜在问题**：如果第一次排程的结果已发布到MES，MES已按4/23的换模计划执行，但第二次排程后4/23的换模计划被删除（原子替换），可能导致MES执行异常。

### 问题8：第二次排程的 inheritedPlanQty 扣减是否正确

**根因代码**（ScheduleAdjustHandler:255-259）：
```java
int inheritedPlanQty = Math.max(0, context.getInheritedPlanQtyMap()
    .getOrDefault(plan.getMaterialCode(), 0));
dto.setWindowPlanQty(windowPlanQty);
dto.setPendingQty(Math.max(0, windowPlanQty - inheritedPlanQty) + carryForwardQty);
```

`inheritedPlanQty` 从 `context.getInheritedPlanQtyMap()` 获取，由 `RollingScheduleHandoffService` 在S4.2阶段写入。该值应该是重叠窗口内已继承的班次计划量之和。

**验证**：
- 3302001884 的 inheritedPlanQty 应该 = 第一次排程中班次5+6+7+8的量 = 22×4 = 88
- 但第二次排程 3302001884 的 pendingQty 需要确认

如果 inheritedPlanQty 计算正确，那么 pendingQty = windowPlanQty - inheritedPlanQty + carryForwardQty 应该反映真实的"新增待排量"。

## 五、实施方案

### 5.1 P1：余量计算需扣减前次排程已安排计划量

**当前逻辑**：
```
surplusQty = TOTAL_QTY - 月累计完成量(截至T-1日)
```

**问题**：前次排程已安排的计划量（尚未完成的部分）未被扣减，导致余量偏大。

**修改方案**：
```
surplusQty = TOTAL_QTY - 月累计完成量(截至T-1日) - 前次排程已安排未完成量
```

**实施步骤**：

1. 在 `LhScheduleContext` 中新增字段 `previousSchedulePlannedQtyMap`（Map<String, Integer>），记录前次排程各物料的已安排计划量

2. 在 `ScheduleAdjustHandler.adjustPreviousSchedule()` 方法中，计算前次排程已安排未完成量：
```java
// 前次排程已安排计划量（仅统计重叠窗口外的班次，重叠窗口内的由继承逻辑处理）
int previousPlannedQty = resolvePreviousPlannedQty(context, result);
context.getPreviousSchedulePlannedQtyMap()
    .merge(result.getMaterialCode(), previousPlannedQty, Integer::sum);
```

3. 在 `calculateSurplusQty()` 中扣减前次排程已安排量：
```java
int previousPlannedNotFinished = Math.max(0,
    context.getPreviousSchedulePlannedQtyMap()
        .getOrDefault(materialCode, 0) - finishedQty);
return new SurplusCalculation(Math.max(0, totalPlanQty - finishedQty - previousPlannedNotFinished));
```

**需要注意**：
- 前次排程已安排量的统计范围需要区分：继承窗口内的班次由继承逻辑处理，不应重复扣减
- carryForwardQty（欠产传导量）的交互：如果前次排程计划量中已包含欠产传导，需要避免双重扣减

**待确认**：前次排程已安排量的具体统计范围——是前次排程的完整8班，还是仅非重叠窗口的班次？

### 5.2 P2：DAILY_PLAN_QTY 取值修正

**当前逻辑**：
```java
dto.setDailyPlanQty(plan.getDayVulcanizationQty() != null ? plan.getDayVulcanizationQty() : 0);
```
取自月计划的 `DAY_VULCANIZATION_QTY` 字段。

**问题**：`DAY_VULCANIZATION_QTY` 是日均硫化量（如33），而非8班班次计划量总和。

**修改方案**：

方案A（推荐）：DAILY_PLAN_QTY 取当前排程窗口班次计划量总和
```java
int dailyPlanQty = ShiftCapacityResolverUtil.resolveDailyPlanQty(
    plan, context.getScheduleWindowShifts());
dto.setDailyPlanQty(dailyPlanQty);
```

方案B：DAILY_PLAN_QTY 取8班班次计划量总和
```java
int dailyPlanQty = ShiftFieldUtil.sumAllShiftPlanQty(plan);
dto.setDailyPlanQty(dailyPlanQty);
```

**待确认**：DAILY_PLAN_QTY 的业务含义是"日计划量"还是"本次排程安排量"？这将决定选择方案A还是方案B。

### 5.3 P3：班次分布优化

**问题**：第二次排程中 3302002531 被排到最后一个班次（班次8），而 3302001884 占据了班次2-5。

**根因**：新增排产时，SKU优先级排序决定了排产顺序。3302001884 优先级高于 3302002531，因此先分配到较好的班次。

**优化方案**：
1. 在排程算法中增加"滚动衔接连续性"约束：如果前次排程中某物料在特定班次生产，本次排程应尽量保持相同班次
2. 增加班次偏好权重：续作SKU优先分配到前次排程的班次位置

**实施步骤**：
1. 在 `SkuScheduleDTO` 中新增字段 `preferredShiftIndex`（偏好班次索引）
2. 在 `RollingScheduleHandoffService` 中，为继承SKU设置偏好班次
3. 在 `NewSpecProductionStrategy` 的班次分配逻辑中，优先满足偏好班次

### 5.4 P4：模具交替计划MES冲突防护

**问题**：原子替换删除已发布排程的模具交替计划，可能导致MES执行异常。

**修改方案**：
1. 在 `SchedulePersistenceService.replaceScheduleAtomically()` 中，增加MES发布状态检查
2. 已发布到MES的排程结果不允许原子替换，应报错提示
3. 如需替换，需先撤回MES发布

**实施步骤**：
```java
// 在原子替换前检查MES发布状态
List<LhScheduleResult> existingResults = scheduleResultMapper.selectList(
    new QueryWrapper<LhScheduleResult>()
        .eq("SCHEDULE_DATE", targetDate)
        .eq("FACTORY_CODE", factoryCode)
        .eq("IS_DELETE", 0));
boolean hasReleased = existingResults.stream()
    .anyMatch(r -> "1".equals(r.getReleaseStatus()));
if (hasReleased) {
    throw new ScheduleException("排程结果已发布到MES，不允许重新排程，请先撤回");
}
```

### 5.6 P6：未排产结果与排程结果并存

**说明**：这是设计行为，不是bug。当SKU的部分待排量成功分配机台、部分因无可用机台失败时，成功部分写入排程结果表，失败部分写入未排产结果表。两表的数据可以同时存在。

**优化建议**：在未排产结果中增加"已排产量"字段，方便对比。

## 六、实施优先级与依赖关系

```
P1(余量扣减) ──→ P3(班次分布优化)
    │                    │
    ↓                    ↓
P2(DAILY_PLAN_QTY)   P5(滚动衔接验证)
    │                    │
    ↓                    ↓
P4(MES冲突防护) ←───────┘
```

## 七、合计余量调整后的影响验证

### 调整内容
将 `loadMaterialMonthFinishedQty` 的截止日期从 `targetDate`（T+2）改为 `scheduleDate - 1`（T-1）。

### 验证结果
以 3302002531 为例：
- 月计划 TOTAL_QTY = 140
- 日完成量：4/20=44, 4/21=66，截至4/23(T-1日)累计完成量=110（4/22和4/23无数据）
- surplusQty = 140 - 110 = 30

**调整前**（截至targetDate=4/25）：同样因为没有4/22-4/25的完成量数据，结果也是30。因此在当前测试数据下，调整前后余量无差异。

**调整生效场景**：当 T日~targetDate 之间存在完成量数据时，调整后 surplusQty 会变大（不再扣减这几天的完成量）。例如：
- 如果4/23有完成量50，调整前 surplusQty = 140 - 160 = 0，调整后 surplusQty = 140 - 110 = 30
- 调整后更符合业务期望：T日当天和之后的完成量不应影响本次排程的余量计算

### 调整对系统的影响
1. **收尾判定**：surplusQty可能变大，部分SKU的收尾判定会延后
2. **目标排产量**：surplusQty变大，TargetScheduleQtyResolver的上限变大
3. **排程结果IS_END**：更难标记为收尾
4. **已验证**：在当前测试数据下，调整前后排程结果一致
