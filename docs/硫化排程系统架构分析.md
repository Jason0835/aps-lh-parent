# 硫化排程系统架构深度分析 OpenCode

## 一、业务流程分析

### 1.1 核心业务概念

硫化排程是轮胎生产的关键环节，将月生产计划分解为机台级日的生产指令。核心业务域包括：

| 概念 | 业务含义 |
|------|---------|
| **月计划** | 从 `FactoryMonthPlanProductionFinalResult` 表获取，按 SKU（物料编码）维护 day1~day31 每日计划量 |
| **日计划额度** | 排程窗口（T日→目标日）内每天的计划量，按 `day1/day2/day3` 映射到具体日期 |
| **机台** | 硫化机台，分为普通机台和单控机台（L/R 独立），每个机台有班产产能、可用窗口 |
| **模具** | 每个 SKU 关联一组模具（`T_MDM_SKU_MOULD_REL`），模具必须在同胎胚的机台间分配 |
| **胎胚** | 半成品，多个 SKU 可共用同一胎胚，胎胚库存需按标准产能占比分摊到各 SKU |
| **续作 (Continuous)** | MES 在机物料继续生产，优先在原机台延续，涉及收尾判定和换活字块衔接 |
| **新增 (New Spec)** | 需要换模上机的新规格，需进行机台匹配、模具分配、首检、班次分配 |
| **换模** | 切换模具需要预热水等换模时间，有每日总量限制（早班≤8，中班≤7，总≤15） |
| **换活字块** | 同模具内切换活字块，耗时不换模，但需要衔接时间 |
| **收尾 (Ending)** | SKU 剩余排量在窗口内可完成时标记为收尾，触发换模计划和降模释放 |
| **欠产/超产传导** | 前日实际完成与计划的偏差净值传导到当日 |
| **开停产** | 工厂级开产/停产班次控制产能比例（停产 Day-3: 30%, Day-2: 60%, Day-1: 90%） |

### 1.2 排程六阶段

S4.1～S4.6 代表了硫化排程的完整生命周期：

| 阶段 | 作用 | 输入 | 输出 |
|------|------|------|------|
| S4.1 前置校验 | 校验 MES 发布状态、排程窗口是否跨月、生成批次号 | 请求参数 | 批次号 / 异常拒绝 |
| S4.2 数据初始化 | 加载班次配置、月计划、机台、模具、MES在机、胎胚库存等 | 工厂+日期 | 机台运行态、班次窗口 |
| S4.3 SKU 归集 | 欠产传导、构建 `SkuScheduleDTO`、标记收尾/续作/新增 | 月计划+库存+前批次结果 | 续作列表、新增列表、日计划账本 |
| S4.4 续作排产 | 优先处理在机物料、收尾、换活字块、班次分配、降模 | 续作列表 | 续作排程结果 |
| S4.5 新增排产 | 遍历新增 SKU、机台匹配、换模均衡、首检、班次分配 | 新增列表 | 新增排程结果 |
| S4.6 结果保存 | 后置校验、换模计划生成、工单号、原子替换目标日结果 | 全部结果 | 持久化 + 事件发布 |

### 2.2 LhScheduleContext 的作用与状态变化

`LhScheduleContext`（`LhScheduleContext.java:54`）是贯穿整个排程生命周期的**可变数据总线**，承载以下角色：

1. **输入数据容器**：排程参数（工厂、目标日、T日、批次号）、配置快照（`LhScheduleConfig`）
2. **基础数据承载**：月计划、机台、模具、胎胚库存、MES在机、工作日历等 Map
3. **中间计算结果**：续作/新增 SKU 列表、结构分组、欠产传导净值
4. **机台运行态**：`machineScheduleMap`（每机台的排程时间线）、`machineShiftCapacityMap`（各班次剩余产能）
5. **输出结果集合**：排程结果、未排结果、模具交替计划、过程日志
6. **流程控制**：中断标识 `interrupted`、当前步骤 `currentStep`、校验错误集合

**状态变化时序**：

```
S4.1 完成后：batchNo ✓, scheduleConfig ✓
S4.2 完成后：monthPlanList ✓, machineInfoMap ✓, machineScheduleMap ✓, shiftRuntimeStateMap ✓
S4.3 完成后：structureSkuMap ✓, continuousSkuList ✓, newSpecSkuList ✓, carryForwardQtyMap ✓
S4.4 完成后：scheduleResultList [续作结果], continuousSkuList 部分消费
S4.5 完成后：scheduleResultList [续作+新增], newSpecSkuList 部分消费 → unscheduledResultList
S4.6 完成后：mouldChangePlanList ✓, processLogList ✓, 数据已落库
```

## 三、设计模式应用

### 3.1 模板方法模式 (Template Method Pattern)

**核心类**：`AbsLhScheduleTemplate`（`AbsLhScheduleTemplate.java:32`）

```java
public final LhScheduleResponseDTO execute(LhScheduleContext context) {
    doPreValidation(context);      // 抽象方法
    doDataInitialization(context);  // 抽象方法
    doAdjustAndGather(context);     // 抽象方法
    doContinuousProduction(context);// 抽象方法
    doNewSpecProduction(context);   // 抽象方法
    doResultValidationAndSave(context); // 抽象方法
}
```

- `execute()` 是 `final` 方法，定义了不可变的 S4.1→S4.6 骨架
- 具体实现 `LhScheduleTemplateImpl` 将各步骤委托给对应 Handler
- **优势**：保证排程步骤顺序绝不改变，新增步骤只需在抽象类中插入，所有子类自动遵循

### 3.2 装饰器模式 (Decorator Pattern)

**核心接口**：`IScheduleExecutor`（`IScheduleExecutor.java:11`）
**抽象装饰器**：`AbsScheduleExecutorDecorator`（`AbsScheduleExecutorDecorator.java:12`）
**配置组装**：`ScheduleExecutorConfig`（`ScheduleExecutorConfig.java:18`）

装饰器链组装顺序：
```java
// ScheduleExecutorConfig.decoratedScheduleExecutor()
IScheduleExecutor executor = defaultScheduleExecutor;          // 核心执行
executor = new LoggingScheduleDecorator(executor);             // 日志层
executor = new PerformanceMonitorDecorator(executor);          // 性能监控层（最外层）
```

调用顺序：`PerformanceMonitor → Logging → Default → AbsLhScheduleTemplate`

- **优势**：横切关注点（日志、性能）与核心业务解耦，新增监控层不改核心代码

### 3.3 策略模式 (Strategy Pattern)

**核心接口列表**（`engine/strategy/` 目录下共11个策略接口）：

| 接口 | 职责 |
|------|------|
| `IProductionStrategy` | 排产策略（续作/新增），含收尾、班次分配、降模等方法 |
| `ISkuPriorityStrategy` | SKU 优先级排序策略 |
| `IMachineMatchStrategy` | 机台匹配策略（收尾时间、规格、英寸、胶囊共用性排序） |
| `IMouldChangeBalanceStrategy` | 换模均衡策略 |
| `IFirstInspectionBalanceStrategy` | 首检均衡策略 |
| `ICapacityCalculateStrategy` | 产能和开产时间计算 |
| `IEndingJudgmentStrategy` | 收尾判定策略 |
| `ITrialProductionStrategy` | 试制策略 |
| `IInsertOrderStrategy` | 插入排序策略 |
| `IProductionShutdownStrategy` | 开停产策略 |
| `ITypeBlockProductionStrategy` | 换活字块策略 |

**工厂模式 + 策略自注册**：`ScheduleStrategyFactory`（`ScheduleStrategyFactory.java:45`）

```java
@PostConstruct
public void init() {
    for (IProductionStrategy strategy : productionStrategies) {
        registerProductionStrategy(strategy); // 按 strategyType 自动注册
    }
}
```

- `IProductionStrategy.getStrategyType()` 返回 `"01"`（续作）或 `"02"`（新增）
- Spring 自动注入所有 `IProductionStrategy` 实现，启动时自动注册到 `productionStrategyCache`
- **优势**：新增排产策略只需实现接口并声明 `getStrategyType()`，无需修改工厂代码（开闭原则）

### 3.4 责任链模式 (Chain of Responsibility)

**核心类**：`DataValidationChain`（`DataValidationChain.java:34`）

- 注入全部 `IDataValidator` 实现，按 `(group, order)` 排序并分组
- 支持两种策略：
  - `COLLECT_ALL`：组内全部执行，收集所有错误后返回
  - `FAIL_FAST`：遇首个错误即短路退出
- 校验器可通过配置开关 `DataValidatorProperties` 动态禁用
- **优势**：校验规则可插拔，不影响主排程流程

### 3.5 观察者模式 (Observer Pattern)

**核心类**：
- `ScheduleEvent`：事件对象（支持 `completed`、`failed`、`published` 三种事件）
- `ScheduleEventPublisher`：发布器，遍历所有 `IScheduleEventListener` 并分发事件
- `IScheduleEventListener`：监听器接口，通过 `supports(EventTypeEnum)` 判断是否处理

当前在 `ResultValidationHandler.doHandle()` 中保存完成后发布 `ScheduleEvent.completed(context)` 事件。

- **优势**：排程完成后的通知逻辑（如通知 MES、更新缓存等）可与核心流程解耦

### 3.6 外观模式 (Facade Pattern)

- `ILhScheduleService` 接口 + `LhScheduleServiceImpl` 实现作为外观，封装了：
  - 上下文构建
  - 分布式锁获取/释放
  - 排程执行器调用
  - 异常统一处理
- Controller 只需调用一个方法，不感知内部复杂性

### 4.2 排程锁机制和并发控制

**实现类**：`ScheduleExecutionGuard`（`ScheduleExecutionGuard.java:27`）

```java
// 锁 Key 格式
APS:LH:SCHEDULE:LOCK:{factoryCode}:{targetDate(yyyy-MM-dd)}

// 获取锁：Redis SET NX EX
setIfAbsent(lockKey, token, 120L, TimeUnit.MINUTES)

// 释放锁：Lua 脚本原子操作
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else return 0 end
```

关键设计点：
- **按工厂+目标日维度加锁**：同一工厂同一天串行执行
- **锁令牌 UUID**：防止误释放他人持有的锁
- **120 分钟 TTL**：防止死锁，正常流程在 `finally` 中释放
- **获取失败抛 ScheduleException**：由 Service 层统一返回失败响应

### 4.3 排程结果的原子替换与数据一致性保证

**实现类**：`SchedulePersistenceService`（`SchedulePersistenceService.java:98`）

```java
@Transactional(rollbackFor = Exception.class)
public void replaceScheduleAtomically(LhScheduleContext context) {
    // 1. 二次校验 MES 发布状态（防止保存瞬间被发布）
    // 2. 查询同目标日旧排程结果 → 收集旧批次号
    // 3. 删除同工厂同目标日旧排程结果（scheduleResultMapper.delete）
    // 4. 删除旧未排结果（unscheduledResultMapper.delete）
    // 5. 删除旧模具交替计划（mouldChangePlanMapper.delete）
    // 6. 按旧批次号删除旧过程日志（processLogMapper.delete）
    // 7. 补齐新结果审计字段和班次收尾标记
    // 8. 批量 insert 新排程结果、未排结果、模具计划、过程日志
}
```

一致性保证机制：
1. **单事务**：所有删除 + 插入在同一事务中，全部成功或全部回滚
2. **先删后插**：以目标日为单位整批替换，避免新旧结果共存
3. **二次校验**：保存前再次检查 `isRelease=1`，防止并发发布导致覆盖
4. **逻辑删除**：旧结果标记 `isDelete=1`（需查看具体删除逻辑）
5. **分布式锁**：同工厂同目标日的排程执行本身是串行的

## 六、总结

该硫化排程系统的架构设计体现了以下工程思想：

1. **分层清晰**：Controller（接口）→ Service（编排）→ Template（步骤骨架）→ Handler（步骤实现）→ Strategy（算法细节）
2. **多模式组合**：模板方法定流程、装饰器做横切、策略做算法可替换、责任链做校验可插拔、工厂做策略注册、观察者做事件通知
3. **上下文贯穿**：`LhScheduleContext` 作为可变数据总线，避免参数传递地狱，同时约束各步骤的读写边界
4. **并发安全**：Redis 分布式锁 + 事务原子替换，保证同目标日排程结果的一致性
5. **可扩展性**：新增排程规则只需实现策略接口 + 自动注册，不影响主流程
