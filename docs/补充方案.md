## 推荐最终补充方案

### 核心原则

把原来的：

```text
SKU 在窗口内一次性排 targetScheduleQty
```

改成：

```text
SKU 按日期生成多个 dailyDemand
每个 dailyDemand 在当天班次内排
当天一台机台排不完，再拆到当天多台机台
当天多台仍排不完，形成当天未排/欠产
```

---

## 建议的数据结构调整

建议不要只在 `SkuScheduleDTO` 上放一个总的 `remainingScheduleQty`，而是增加日维度结构，例如：

```java
public class SkuDailyScheduleDemandDTO implements Serializable {

    private String materialCode;

    private LocalDate scheduleDate;

    /**
     * 月计划当天计划量，例如 day7/day8/day9
     */
    private Integer dayPlanQty;

    /**
     * 当天已继承/已锁定/已排产量
     */
    private Integer inheritedQty;

    /**
     * 当天欠产结转量，可按项目现有欠产规则决定是否叠加
     */
    private Integer carryForwardQty;

    /**
     * 当天目标排产量
     */
    private Integer targetQty;

    /**
     * 当天剩余未排量
     */
    private Integer remainingQty;
}
```

`SkuScheduleDTO` 可以保留 SKU 主信息，但排产时不要只用一个总 `targetScheduleQty`，而是挂一个：

```java
private List<SkuDailyScheduleDemandDTO> dailyDemandList;
```

## 目标量计算建议

### 非收尾场景

每天严格按当天月计划排：

```text
dailyTargetQty = min(dayPlanQty + carryForwardQty - inheritedQty, monthRemainQty)
```

如果项目要求“欠产不挤占当天计划”，也可以改成：

```text
dailyTargetQty = dayPlanQty - inheritedQty
```

欠产单独排在欠产池中，不和当天计划混排。

### 收尾场景

只有当 SKU 月计划余量可以在排程窗口内通过多机台合计产能完成时，才允许胎胚库存上调：

```text
if canFinishInWindow:
    totalTargetQty = max(monthRemainQty, embryoStockQty)
else:
    totalTargetQty = monthRemainQty
```

但落地时仍然要分配到日期：

```text
优先满足 day7/day8/day9 原计划量
收尾上调出来的额外量，建议只放到最后一天或收尾日期
```

比如：

```text
day7 = 100
day8 = 80
day9 = 120
胎胚库存收尾需排 350
原窗口计划合计 300
额外 50 建议落到 day9 收尾量
```

不要把额外 50 平摊到 day7/day8，否则会破坏日计划严格性。

## 多机台拆量逻辑建议

按“日期 + SKU”循环，而不是只按 SKU 循环：

```text
for sku in skuList:
    for dailyDemand in sku.dailyDemandList:
        remainingQty = dailyDemand.targetQty

        while remainingQty > 0:
            machine = selectBestCandidateMachine(sku, scheduleDate)
            if machine == null:
                break

            machineAvailableQty = calcMachineAvailableQty(machine, scheduleDate, sku)
            scheduleQty = min(remainingQty, machineAvailableQty)

            if scheduleQty <= 0:
                exclude machine
                continue

            buildScheduleResult(sku, machine, scheduleDate, scheduleQty)

            remainingQty -= scheduleQty

        if remainingQty > 0:
            addDailyUnscheduledResult(sku, scheduleDate, remainingQty)
```

关键点是：

```text
machineAvailableQty 必须只计算当天班次可用产能
buildScheduleResult 只能写当天对应班次的产量
remainingQty 是当天剩余量，不是窗口总剩余量
```

## 验证建议

### 9.1 单元测试验证

执行步骤9中的单元测试，覆盖所有边界场景。

### 9.2 对比验证

在相同排程输入条件下，对比改造前后的排程结果：

1. **不涉及多机台的SKU**：排产结果应完全一致
2. **涉及多机台的SKU**：
   - 改造前：只在一台机台排产，可能有未排量
   - 改造后：多台机台排产，未排量应减少或为0

### 9.3 日志验证

检查以下日志输出：

1. `SKU多机台排产计算` - 确认窗口计划量、月计划余量、胎胚库存、收尾判断正确
2. `SKU多机台拆量` - 确认每台机台分配量正确，剩余量递减
3. `SKU非收尾场景跳过胎胚库存上调` - 确认非收尾场景不触发上调
4. `SKU多机台收尾产能计算` - 确认多机台合计产能计算正确

### 9.4 数据验证

验证以下数据一致性：

1. SKU总排产量 = 各机台排产量之和
2. SKU总排产量 <= 月计划余量
3. SKU总排产量 <= 窗口计划量（非收尾场景）
4. SKU总排产量 <= 多机台可用总产能
5. 每台机台的排产量 <= 该机台可用产能
6. 同一机台同一班次不被多个SKU重复占用
7. 月计划扣减量 = SKU总排产量


特别是多机台拆量、非收尾不上调、收尾判断多机台这些验证点



> **不能把排程主流程改成“每天拆开排一次”。**
> 硫化排程整体仍应保持：**一个 SKU 在机台上连续排 8 个班次**。
> 但在这个连续 8 班的过程中，需要增加 **日计划口径的约束、扣减和校验**。

也就是说：

```text
错误理解：
按 day7、day8、day9 把 SKU 拆成 3 个独立排程任务，每天重新选机、重新排。

正确理解：
SKU 仍按连续 8 个班次排，只是每个班次产量落到对应日期后，
要校验当天累计量是否符合 day7/day8/day9。
```

## 为什么不能简单按天拆开排

如果按天拆，会破坏硫化排程的几个核心规则：

1. **连续生产被打断**
   一个 SKU 原本在一台机台上连续跑 8 个班次，如果按天拆，可能变成 5月7日排一段、5月8日重新排一段、5月9日再排一段。

2. **换模/换活字块逻辑会被放大**
   每天拆开后，系统可能误以为每天都是一次新的排产动作，导致换模、首检、换活字块判断复杂化。

3. **续作衔接会变乱**
   现有项目里续作、收尾、换模、新规格排产大概率都是围绕“机台连续状态”设计的。如果按日期强拆，会让机台状态变得不自然。

4. **不符合“一个 SKU 连续排 8 班”的业务习惯**
   硫化更像是机台连续作业，不是每天独立派工。

所以，**不能把主排程逻辑改成按天外层循环。**

---

## 更合理的做法：连续 8 班 + 日计划桶控制

推荐改成这个口径：

```text
SKU 仍然一次性按连续 8 个班次排产；
每个班次产量根据班次所属日期，累计到对应日期的日计划桶；
每个日期的累计排产量不能超过该日期月计划 dayN；
如果当天计划量已经满足，则后续班次不能继续占用当天计划量。
```

比如排程窗口：

```text
2026-05-07 ~ 2026-05-09

day7 = 100
day8 = 80
day9 = 120
```

SKU 连续排 8 个班次：

```text
第1班：2026-05-07 早班
第2班：2026-05-07 中班
第3班：2026-05-07 夜班
第4班：2026-05-08 早班
第5班：2026-05-08 中班
第6班：2026-05-08 夜班
第7班：2026-05-09 早班
第8班：2026-05-09 中班
```

那么系统不是先拆成 3 个任务，而是连续排的时候维护：

```text
2026-05-07 已排量 <= day7
2026-05-08 已排量 <= day8
2026-05-09 已排量 <= day9
```

## 关键区别

### 不推荐

```java
for (date in scheduleWindow) {
    for (sku in skuList) {
        scheduleSkuOnDate(sku, date);
    }
}
```

这个会变成“按天排 SKU”，违背连续 8 班。

---

### 推荐

```java
for (sku in skuList) {
    targetQty = resolveWindowTargetQty(sku);

    while (remainingQty > 0) {
        machine = selectMachine(sku);

        result = scheduleContinuousEightShifts(sku, machine);

        // 按结果中的班次日期，把产量分摊/累计到对应 day7/day8/day9
        checkAndDeductDailyPlanBucket(result);
    }
}
```

也就是：

```text
排程动作：仍然按 SKU 连续 8 班
控制口径：增加日期维度的日计划桶
```

---

## 多机台拆量也应该保持这个原则

一个 SKU 一台机台排不完时，可以拆到多台机台，但每台机台上仍然是连续排：

```text
SKU A 总需求 240

机台 M1 连续排 8 班，排 160
机台 M2 连续排 8 班，排 80
```

而不是：

```text
5月7日 SKU A 排 M1、M2
5月8日 SKU A 重新排 M3
5月9日 SKU A 重新排 M4
```

更准确的逻辑应该是：

```text
SKU A 按窗口目标量进入排程
M1 连续排，排产结果落到 day7/day8/day9
如果 M1 产能不够，再找 M2 连续排
M2 的结果也落到 day7/day8/day9
最后校验每一天的累计量是否符合月计划
```

---


## 所以我建议最终口径这样定

更适合硫化排程项目的表述是：

```text
硫化排程主流程仍保持“一个 SKU 连续排 8 个班次”的方式，不改为按天拆开独立排产。

本次优化不是改变连续排产模式，而是在连续排产过程中增加“日计划桶”控制：
1. 排程窗口内的目标量仍可由 day7 + day8 + day9 汇总得到；
2. 但排程结果落班时，需要按班次所属日期累计到对应 dayN；
3. 每个日期的累计排产量原则上不能超过该日期月计划量；
4. 如果某日期一台机台在连续 8 班中的产能不足，可以允许同一 SKU 在其他机台上继续连续排产，但结果仍需落到对应日期；
5. 非收尾场景下，不允许因为胎胚库存把某天计划量上调；
6. 收尾场景下，才允许在窗口总目标量上调到胎胚库存，但上调量建议落到收尾日期或最后一个可排日期；
7. 如果要求日计划严格等于 dayN，需要支持班次内产量裁剪，否则整班产能可能导致当天超计划。
```

---

## 对前面方案的修正建议

我会把之前的“按日拆量控制”修正为：

> **不是按日期拆成多个独立排程任务，而是在 SKU 连续 8 班排产过程中，引入日计划桶进行扣减、封顶和校验。**

这样就不违背“一个 SKU 连续排 8 个班次”的整体排产方式。
