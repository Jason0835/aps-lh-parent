# 可配置规则引擎使用指南

## 一、概述

可配置规则引擎允许您通过数据库动态调整排程规则，无需修改代码或重启系统。所有业务规则从硬编码的常量类 `LhScheduleConstant` 抽取到数据库表 `T_LH_PARAMS` 中。

## 二、架构设计

### 2.1 组件关系

```
┌─────────────────────────┐
│   策略类/Handler 类      │
│  (ContinuousProduction- │
│   Strategy 等)          │
└───────────┬─────────────┘
            │ 注入
            ▼
┌─────────────────────────┐
│  IScheduleRuleEngine    │  ◄─── 接口定义
│  (DatabaseSchedule-     │  ◄─── 数据库实现
│   RuleEngine)           │
└───────────┬─────────────┘
            │ 读取
            ▼
┌─────────────────────────┐
│   T_LH_PARAMS 表        │  ◄─── 规则配置存储
└───────────┬─────────────┘
            │ 缓存
            ▼
┌─────────────────────────┐
│   本地缓存 (5 分钟)       │  ◄─── 性能优化
└─────────────────────────┘
```

### 2.2 优先级策略

按照您的要求，系统采用以下优先级:

**数据库配置 > 上下文参数 > 常量默认值**

- **数据库配置**: 从 `T_LH_PARAMS` 表读取，支持动态修改
- **上下文参数**: 从 `LhScheduleContext.getParamIntValue()` 读取，支持临时覆盖
- **常量默认值**: 从 `LhScheduleConstant` 读取，作为最终兜底

## 四、使用方法

### 4.1 在代码中使用规则引擎

#### 方式 1: 直接注入规则引擎 (推荐)

```java
@Component
public class YourStrategy {
    
    @Resource
    private IScheduleRuleEngine ruleEngine;
    
    public void doSomething(LhScheduleContext context) {
        String factoryCode = context.getFactoryCode();
        
        // 获取规则值
        int mouldChangeLimit = ruleEngine.getDailyMouldChangeLimit(factoryCode);
        double efficiency = ruleEngine.getShiftEfficiencyFactor(factoryCode, "MORNING");
        
        // 使用规则值进行业务逻辑
        if (currentMouldChangeCount >= mouldChangeLimit) {
            // 已达换模上限
            return;
        }
    }
}
```

#### 方式 2: 通过 LhScheduleTimeUtil 间接使用

```java
// LhScheduleTimeUtil 内部会从 context 获取 factoryCode
int mouldChangeHours = LhScheduleTimeUtil.getMouldChangeTotalHours(context);
Date startTime = LhScheduleTimeUtil.addHours(endingTime, mouldChangeHours);
```

### 4.2 在数据库中配置规则

#### 新增规则配置

```sql
INSERT INTO T_LH_PARAMS (FACTORY_CODE, PARAM_CODE, PARAM_VALUE, PARAM_NAME, REMARK) 
VALUES ('F001', 'DAILY_MOULD_CHANGE_LIMIT', '20', '每日换模上限', '提升换模能力到 20 次');
```

#### 修改规则配置

```sql
UPDATE T_LH_PARAMS 
SET PARAM_VALUE = '25' 
WHERE FACTORY_CODE = 'F001' AND PARAM_CODE = 'DAILY_MOULD_CHANGE_LIMIT';
```

#### 删除规则配置 (逻辑删除)

```sql
UPDATE T_LH_PARAMS 
SET IS_DELETE = 1 
WHERE FACTORY_CODE = 'F001' AND PARAM_CODE = 'DAILY_MOULD_CHANGE_LIMIT';
```

#### 查询规则配置

```sql
-- 查询某工厂的所有规则
SELECT * FROM T_LH_PARAMS 
WHERE FACTORY_CODE = 'F001' AND IS_DELETE = 0 
ORDER BY PARAM_CODE;

-- 查询特定规则
SELECT * FROM T_LH_PARAMS 
WHERE PARAM_CODE = 'DAILY_MOULD_CHANGE_LIMIT' AND IS_DELETE = 0;
```

### 4.3 清除缓存 (可选)

规则引擎会自动每 5 分钟刷新缓存，如需立即生效可手动清除:

```java
@Resource
private DatabaseScheduleRuleEngine ruleEngine;

// 清除某工厂的缓存
ruleEngine.clearCache("F001");

// 清除所有缓存
ruleEngine.clearCache(null);
```

## 五、最佳实践

### 5.1 规则配置建议

1. **首次部署**: 执行 `rule_engine_init.sql` 初始化脚本，导入默认配置
2. **灰度测试**: 先为单个工厂配置规则，验证无误后再推广到其他工厂
3. **变更记录**: 建立规则变更审批流程，记录每次修改的原因和时间
4. **监控告警**: 对关键规则 (如换模上限) 设置监控，异常修改时告警

### 5.2 性能优化

1. **缓存机制**: 规则引擎内置 5 分钟缓存，避免频繁查询数据库
2. **按需加载**: 只在首次访问时加载规则，后续直接使用缓存
3. **批量操作**: 避免在循环中多次调用规则引擎，提前缓存到局部变量

```java
// ❌ 不推荐：循环内多次调用
for (SkuScheduleDTO sku : skus) {
    if (count < ruleEngine.getDailyMouldChangeLimit(factoryCode)) {
        // ...
    }
}

// ✅ 推荐：提前缓存
int limit = ruleEngine.getDailyMouldChangeLimit(factoryCode);
for (SkuScheduleDTO sku : skus) {
    if (count < limit) {
        // ...
    }
}
```

### 5.3 故障排查

#### 问题 1: 规则未生效

**检查步骤**:
1. 确认 `T_LH_PARAMS` 表中记录存在且 `IS_DELETE = 0`
2. 确认 `FACTORY_CODE` 匹配
3. 等待 5 分钟让缓存自动刷新，或手动清除缓存
4. 查看日志是否有"参数解析失败"的警告

#### 问题 2: 规则引擎未注入

**检查步骤**:
1. 确认类上有 `@Component` 或 `@Service` 注解
2. 确认字段上有 `@Resource` 或 `@Autowired` 注解
3. 确认包扫描路径包含 `com.zlt.aps.lh.engine.rule`

#### 问题 3: 参数类型错误

**检查步骤**:
1. 确认 `PARAM_VALUE` 格式正确 (整数、小数)
2. 查看日志中的 `NumberFormatException` 警告
3. 修正后重新插入或更新记录

## 六、扩展示例

### 6.1 新增规则类型

假设需要新增一个"最小生产批量"规则:

**步骤 1**: 在 `IScheduleRuleEngine` 接口添加方法

```java
/**
 * 获取最小生产批量
 */
int getMinProductionBatchSize(String factoryCode);
```

**步骤 2**: 在 `DatabaseScheduleRuleEngine` 实现方法

```java
@Override
public int getMinProductionBatchSize(String factoryCode) {
    return getIntValue(factoryCode, "MIN_PRODUCTION_BATCH_SIZE", 100);
}
```

**步骤 3**: 在数据库中插入配置

```sql
INSERT INTO T_LH_PARAMS (FACTORY_CODE, PARAM_CODE, PARAM_VALUE, PARAM_NAME, REMARK) 
VALUES ('F001', 'MIN_PRODUCTION_BATCH_SIZE', '150', '最小生产批量', '每次排产的最小批量');
```

**步骤 4**: 在策略中使用

```java
int minBatchSize = ruleEngine.getMinProductionBatchSize(factoryCode);
if (pendingQty < minBatchSize) {
    // 不足最小批量，暂不排产
    return;
}
```

## 七、常见问题 FAQ

### Q1: 修改规则后多久生效？
**A**: 最多 5 分钟。规则引擎每 5 分钟自动刷新缓存。也可手动调用 `clearCache()` 立即生效。

### Q2: 不同工厂可以有不同的规则吗？
**A**: 可以。所有规则方法都接受 `factoryCode` 参数，支持按工厂隔离配置。

### Q3: 如果数据库中没有某个规则配置会怎样？
**A**: 会自动使用 `LhScheduleConstant` 中的默认值，不会影响系统运行。

### Q4: 规则引擎会影响性能吗？
**A**: 几乎无影响。规则引擎使用本地缓存，命中缓存时直接返回，无需数据库查询。

### Q5: 可以在运行时动态添加新规则吗？
**A**: 可以。在数据库中添加记录即可，但需要在代码中定义对应的接口方法和实现。

## 九、总结

可配置规则引擎提供了灵活、高效的规则管理方式:

✅ **动态配置**: 无需修改代码或重启系统  
✅ **工厂隔离**: 支持不同工厂不同的规则  
✅ **性能优化**: 5 分钟缓存机制  
✅ **向后兼容**: 保持与原有代码的兼容性  
✅ **易于扩展**: 新增规则简单快捷  

通过本引擎，您可以轻松实现:
- 根据实际生产能力调整换模上限
- 根据不同班次的效率调整产能计算
- 针对特殊时期 (如检修月) 调整规则
- 快速响应业务变化，提升系统灵活性
