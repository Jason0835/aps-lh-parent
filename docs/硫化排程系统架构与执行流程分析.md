# 硫化排程系统架构与执行流程深度分析

---

## 一、整体架构概览

硫化排程系统采用 **分层架构 + 设计模式组合** 的设计，核心调用链路如下：

```
┌─────────────────────────────────────────────────────────────────────────┐
│  Controller 层:  LhScheduleResultController (HTTP 入口)                  │
│    POST /lhScheduleResult/execute                                        │
│    POST /lhScheduleResult/publish/{batchNo}                              │
└──────────────────────────┬──────────────────────────────────────────────┘
                           │ 委托
┌──────────────────────────▼──────────────────────────────────────────────┐
│  Service 层:     LhScheduleServiceImpl (流程编排 + 边界控制)              │
│    - buildContext()       构建排程上下文                                  │
│    - ScheduleExecutionGuard  获取 Redis 分布式锁                         │
│    - IScheduleExecutor.execute()  委托排程引擎                           │
│    - ScheduleEventPublisher.publish()  发布事件                          │
└──────────────────────────┬──────────────────────────────────────────────┘
                           │ 委托
┌──────────────────────────▼──────────────────────────────────────────────┐
│  Engine 层:      DefaultScheduleExecutor → AbsLhScheduleTemplate        │
│    模板方法骨架控制六步排程执行顺序                                       │
│    S4.1 → S4.2 → S4.3 → S4.4 → S4.5 → S4.6                              │
└──────────────────────────┬──────────────────────────────────────────────┘
                           │ 委托
┌──────────────────────────▼──────────────────────────────────────────────┐
│  Handler 层:     六个 Handler (AbsScheduleStepHandler 子类)              │
│    PreValidation / DataInit / ScheduleAdjust /                          │
│    ContinuousProduction / NewProduction / ResultValidation              │
│    每个 Handler 内部调用策略接口完成具体业务逻辑                            │
└──────────────────────────┬──────────────────────────────────────────────┘
                           │ 委托
┌──────────────────────────▼──────────────────────────────────────────────┐
│  Strategy 层:    11 个策略接口 + 13 个实现类                              │
│    IMachineMatchStrategy / ISkuPriorityStrategy /                       │
│    IMouldChangeBalanceStrategy / ITrialProductionStrategy / ...          │
└─────────────────────────────────────────────────────────────────────────┘
```

**贯穿全部层级的数据总线**：`LhScheduleContext`——承载排程参数、基础数据、中间结果、机台状态和最终输出。

## 二、业务流程详解：六步排程引擎 (S4.1 → S4.6)

### S4.1 前置校验与数据清理 (`PreValidationHandler`)

**核心目的**：确保排程环境合法，为后续计算提供安全基线。

| 子步骤 | 逻辑说明 |
|--------|----------|
| S4.1.1 校验 MES 下发状态 | 查询 `LhScheduleResult.isRelease=1` 的记录，如该目标日已有已发布结果，直接拒绝排程（需先撤销发布）。防止覆盖 MES 已执行的排程。 |
| S4.1.2 校验跨月窗口 | 通过 `MonthPlanDayQtyUtil.isCrossMonthWindow()` 判断排程窗口是否跨月。跨月排程不支持（因为月计划 dayN 口径按月对齐，跨月会导致计划量口径不一致）。 |
| S4.1.3 生成批次号 | 调用 `ILhScheduleResultService.generateNextBatchNo()`，格式 `LHPCyyyyMMdd+流水`，通过 Redis 按工厂+日期维度自增全局分配，24h TTL。 |

### S4.2 基础数据初始化 (`DataInitHandler`)

**核心目的**：加载排程计算所需的全部基础数据，构建机台运行态。

| 子步骤 | 逻辑说明 |
|--------|----------|
| S4.2.1 班次解析 | `ILhShiftConfigService.resolveAndAttachScheduleShifts()` 解析排程窗口内 1~8 班的实际时间窗口（每班含 workDate、开始/结束时间）。无数据库配置时使用默认模板。 |
| S4.2.2 基础数据加载 | `ILhBaseDataService.loadAllBaseData()` 一次性加载：月计划、工作日历、SKU 产能、机台、模具关系、MES 在机信息、胎胚库存、停机计划、清洗计划、特殊物料、定点机台、前批次排程结果等 **25+ 类基础数据**。 |
| S4.2.3 数据校验链 | `DataValidationChain.validateWithResult()` 以责任链模式聚合校验各数据完整性（月计划是否为空、机台是否配置、工作日历是否存在等），收集全部错误后统一失败。 |
| S4.2.4 机台运行态构建 | 为每台硫化机初始化 `MachineScheduleDTO`：MES 在机规格、停机/维修计划、清洗窗口、胶囊使用次数、模台数、宽基胎/芯片胎支持标记、班次可用状态。同时保存 `initialMachineScheduleMap` 快照供回归校验。 |
| 滚动排程衔接 | 非强制重排模式下，`RollingScheduleHandoffService.apply()` 将前批次重叠班次结果继承到本次，推进机台 `estimatedEndTime`、`currentMaterialCode` 等状态。 |

### S4.3 排程调整与 SKU 归集 (`ScheduleAdjustHandler`)

**核心目的**：将月计划按 SKU 维度展开，计算待排目标量，分类为续作/新增两类。

**关键处理**：

- **月计划 SKU 展开**：按 `monthPlanList` 展开每个 SKU 的窗口计划量（day1/day2/day3），结合月余量、日完成量计算待排目标量
- **欠产/超产传导**：前一日欠产/超产净值传导到当日，更新日计划账本
- **胎胚库存分摊**：同胎胚下多个 SKU 按标准产能占比分摊胎胚库存
- **SKU 归集**：按结构名 (`structureName`) 分组，同时按 MES 在机/前批次状态将 SKU 分类为：
  - `continuousSkuList`：续作 SKU（MES 在机相同规格、前批次未收尾等）
  - `newSpecSkuList`：新增 SKU（不在续作列表中的）
- **收尾判断**：多维度收尾判定（结构收尾、SKU 收尾、订单收尾等）

### S4.4 续作规格排产 (`ContinuousProductionHandler`)

**核心目的**：优先消费 MES 在机和滚动继承状态，延续已有生产的 SKU。

**关键处理**：

- **续作排产主流程**：遍历 `continuousSkuList`，为每个续作 SKU 在已在产机台上延续生产
- **换活字块**：同机台同胎胚但不同花纹时，产生换活字块衔接（不停机，只是换活字块）
- **续作收尾**：判断是否需要收尾（达到目标量、胎胚库存不足等场景）
- **续作非收尾挤量**：如果当前窗口内无法完成目标量，继续挤占后续班次
- **续作降模**：同 SKU 多机台降模优化（胶囊使用次数高的机台优先保留）
- **定点机台预留**：定点机台挤量预留切换开始时间，给后续定点新增 SKU 留出换模窗口

消费完毕后，未完成目标量的 SKU 保留到 S4.5 新增链路。

### S4.5 新增规格排产 (`NewProductionHandler`)

**核心目的**：为新增 SKU 分配机台、班次和计划量。

**关键处理**：

- **优先级排序**：`ISkuPriorityStrategy.sortByPriority()` 按发货要求、延误上机、结构收尾日、供应链优先级排序
- **机台匹配**：`IMachineMatchStrategy.matchMachines()` 多层过滤：
  1. 定点机台约束
  2. 寸口/模套/特殊材料/模具硬性匹配
  3. 长时间停机自动换机
  4. 单控/普通机台类型约束
  5. 清洗/保养窗口规避
  6. 多维度排序（胶囊使用次数优先、在产优先、清洗计划最近优先等）
- **选机**：`IMachineMatchStrategy.selectBestMachine()` 从候选机台中选最优
- **换模判断**：新规格与机台当前在产规格不同时产生换模（含换模预热时间、换模时间）
- **首检均衡**：`IFirstInspectionBalanceStrategy` 按日+班次维度均衡首检机台数
- **班次分配**：按机台剩余产能和日计划账本逐班次分配计划量
- **补满逻辑**：最后班次允许补满班产（`shiftFillOverQty` 超排量记账）
- **同 SKU 多机台收口**：多机台并行排产后统一收尾，标记主/辅机台

### S4.6 结果校验与发布保存 (`ResultValidationHandler`)

**核心目的**：保存前最后一道业务防线，生成辅助计划并原子落库。

| 子步骤 | 逻辑说明 |
|--------|----------|
| S4.6.1 后置校验 | 校验排程结果必填字段（machineCode、materialCode、specEndTime、换模时 mouldCode 等），校验计划量口径（严格目标量 SKU 不超排、正式 SKU 超排不超班产范围等） |
| S4.6.2 模具交替计划 | 基于排程结果换模信息生成 `LhMouldChangePlan`：前规格→后规格、换模/换活字块/清洗类型、班次分配，同时追加干冰/喷砂清洗交替计划 |
| S4.6.3 补全工单号 | 排程结果工单号 `LH+yyyyMMdd+流水`，换模工单号 `CHG+yyyyMMdd+流水` |
| S4.6.4 排程顺序 | `IncrSerialGenerator` 按落库顺序生成排程序号 |
| S4.6.5 汇总日志 | 按 SKU+日期汇总日计划完成情况（超排/欠产），输出日计划滚动台账明细 |
| S4.6.5.2 硫化示方历史保护 | 按班次逐班判断是否历史班次，历史班次保留上一轮排程的硫化示方号和类型 |
| S4.6.6 原子持久化 | `SchedulePersistenceService.replaceScheduleAtomically()` 事务内完成：二次校验已发布 → 删除旧结果/未排/换模/日志 → 批量写入新数据 |
| S4.6.7 事件发布 | `ScheduleEventPublisher.publish(ScheduleEvent.completed())` 通知所有监听器排程完成 |

## 三、`LhScheduleContext` 上下文详解

`LhScheduleContext` 是整个排程系统的**可变数据总线**，贯穿全部六步。其字段分为五大类：

### 3.1 排程基本参数

- `factoryCode` / `factoryName`：工厂标识
- `scheduleTargetDate`：排程目标日（请求传入的日期，如 T+2）
- `scheduleDate`：排程窗口起点 T 日（= targetDate - (scheduleDays - 1)）
- `batchNo`：批次号，S4.1 阶段由 Redis 生成
- `scheduleConfig`：本次排程配置快照（不可变，含所有硫化参数）

### 3.2 基础数据（S4.2 加载，25+ Map/List）

月计划、机台、模具、SKU 产能、MES 在机、胎胚库存、工作日历、停机/保养/清洗计划、特殊物料、定点机台等。

### 3.3 中间计算结果（S4.3 归集）

- `structureSkuMap`：SKU 按结构分组
- `continuousSkuList` / `newSpecSkuList`：续作/新增 SKU 列表
- `rollingInheritedScheduleResultList`：滚动继承结果
- `scheduleResultSourceSkuMap`：结果→来源 SKU 映射（IdentityHashMap）

### 3.4 机台分配状态

- `machineScheduleMap`：机台运行态（排程过程中被持续修改）
- `initialMachineScheduleMap`：机台初始状态快照（供回归校验）
- `machineShiftCapacityMap`：各班次剩余产能
- `shiftRuntimeStateMap`：班次运行态（开停产管控）
- `dailyMouldChangeCountMap`：每日换模计数

### 3.5 流程控制

- `interrupted` / `interruptReason`：中断标志
- `currentStep`：当前执行步骤
- `validationErrorList`：校验错误收集
- `priorityTraceMuteDepth`：局部搜索静默深度（抑制非主路径日志）

## 四、设计模式应用分析

### 4.1 模板方法模式 (Template Method Pattern)

**应用位置**：`AbsLhScheduleTemplate` + `LhScheduleTemplateImpl`

```java
// AbsLhScheduleTemplate 定义了不可变的算法骨架
public final LhScheduleResponseDTO execute(LhScheduleContext context) {
    doPreValidation(context);      // S4.1 → PreValidationHandler
    doDataInitialization(context); // S4.2 → DataInitHandler
    doAdjustAndGather(context);    // S4.3 → ScheduleAdjustHandler
    doContinuousProduction(context); // S4.4 → ContinuousProductionHandler
    doNewSpecProduction(context);  // S4.5 → NewProductionHandler
    doResultValidationAndSave(context); // S4.6 → ResultValidationHandler
}
```

**优势**：

- `execute()` 方法使用 `final` 修饰，保证六步执行顺序不可被子类篡改
- 各步骤只暴露 `abstract` 方法，具体实现委托给 Handler，主流程与算法细节解耦
- 统一处理中断响应、异常捕获、耗时日志

### 4.2 策略模式 (Strategy Pattern)

**应用位置**：`engine/strategy` 包

共定义了 **11 个策略接口**，每个有唯一默认实现：

| 策略接口 | 作用 | 实现类 |
|----------|------|--------|
| `IMachineMatchStrategy` | 机台匹配与选择 | `DefaultMachineMatchStrategy` |
| `ISkuPriorityStrategy` | SKU 优先级排序 | `DefaultSkuPriorityStrategy` |
| `IMouldChangeBalanceStrategy` | 换模均衡 | `DefaultMouldChangeBalanceStrategy` |
| `IFirstInspectionBalanceStrategy` | 首检均衡 | `DefaultFirstInspectionBalanceStrategy` |
| `IEndingJudgmentStrategy` | 收尾判定 | `DefaultEndingJudgmentStrategy` |
| `ITrialProductionStrategy` | 试制策略 | `DefaultTrialProductionStrategy` |
| `ITypeBlockProductionStrategy` | 换活字块策略 | `TypeBlockProductionStrategy` |
| `IProductionShutdownStrategy` | 开停产管控 | `DefaultProductionShutdownStrategy` |
| `ICapacityCalculateStrategy` | 产能计算 | `DefaultCapacityCalculateStrategy` |
| `IInsertOrderStrategy` | 插入排序 | `DefaultInsertOrderStrategy` |
| `IProductionStrategy` | 排产执行 | 多个实现 |

**优势**：

- 策略可插拔，新增策略只需新增实现类并注入
- 各策略独立演进，不互相影响
- Handler 中通过 `@Resource` 注入策略接口，运行时可替换实现

### 4.3 装饰器模式 (Decorator Pattern)

**应用位置**：`IScheduleExecutor` + `DefaultScheduleExecutor`

```java
public interface IScheduleExecutor {
    LhScheduleResponseDTO execute(LhScheduleContext context);
}

@Component
public class DefaultScheduleExecutor implements IScheduleExecutor {
    @Resource
    private AbsLhScheduleTemplate scheduleTemplate;
    
    @Override
    public LhScheduleResponseDTO execute(LhScheduleContext context) {
        return scheduleTemplate.execute(context);
    }
}
```

**设计意图**：装饰器层是预留的扩展点。当前 `DefaultScheduleExecutor` 直接委托给模板，但未来可在装饰器层增加：

- 性能监控切面
- 参数校验增强
- 排程前后钩子
- 缓存命中逻辑

接口在 `IScheduleExecutor` 层面暴露，`LhScheduleServiceImpl` 只依赖接口，符合依赖倒置原则。

### 4.4 观察者模式 (Observer Pattern)

**应用位置**：`ScheduleEventPublisher` + `IScheduleEventListener`

```java
// 发布器通过 @Autowired(required = false) 自动收集所有监听器
public void publish(ScheduleEvent event) {
    for (IScheduleEventListener listener : listeners) {
        if (listener.supports(event.getEventType())) {
            listener.onEvent(event);
        }
    }
}
```

**三种事件类型**：

- `SCHEDULE_COMPLETED`：排程完成后通知（如通知 MES 有新排程结果）
- `SCHEDULE_FAILED`：排程失败后通知
- `RESULT_PUBLISHED`：排程发布后通知

**优势**：

- 事件发布与消费解耦
- 每个监听器异常隔离（try-catch 包裹，不影响其他监听器）
- 新增监听器只需实现 `IScheduleEventListener` 并注册为 Spring Bean

### 4.5 责任链模式 (Chain of Responsibility)

**应用位置**：`DataValidationChain` (S4.2 数据校验) + `AbsScheduleStepHandler` 中断串联

**数据校验链**：聚合多个校验器（月计划校验、机台校验、工作日历校验等），收集全部错误后统一失败，而非遇到第一个错误就中断。

**Handler 中断链**：通过 `context.isInterrupted()` 机制串联，每个 Handler 执行前检查中断标志，已中断则跳过后续步骤。

### 4.6 数据总线模式 (Data Bus)

**应用位置**：`LhScheduleContext`

整体排程过程中，所有 Handler 和 Strategy 都读写同一个 Context 对象。所有中间状态都存在 Context 中，避免了复杂的方法参数传递。

## 五、关键组件交互详解

### 5.1 排程锁机制：`ScheduleExecutionGuard`

```java
// 锁 Key 格式: APS:LH:SCHEDULE:LOCK:{factoryCode}:{targetDate}
// 锁 TTL: 120 分钟（默认）

// 获取锁：Redis SETNX + UUID token
lockToken = scheduleExecutionGuard.acquire(factoryCode, targetDate);

// 释放锁：Lua 脚本原子性校验 token 后删除
scheduleExecutionGuard.release(factoryCode, targetDate, lockToken);
```

**设计要点**：

1. **锁粒度**：厂级+目标日级，不同工厂/不同日期可并行排程
2. **互斥保证**：Redis `SETNX` 实现，同一工厂同一目标日串行执行
3. **安全释放**：Lua 脚本保证仅锁持有者可释放，防止 A 的锁被 B 误删
4. **自动过期**：120 分钟 TTL 防止死锁（异常退出或 Redis 故障时自动释放）
5. **异常触发**：获取失败直接抛 `ScheduleException`，外部 catch 后返回 `fail` 响应

**生命周期**：

```
LhScheduleServiceImpl.executeSchedule()
  ├── lockToken = guard.acquire()    // 获取锁
  ├── executor.execute(context)      // 执行排程
  ├── ...                            // 异常处理
  └── finally { guard.release() }    // 保证释放
```

### 5.2 排程事件发布机制：`ScheduleEventPublisher`

```
排程完成 → ScheduleEvent.completed(context)
          → ScheduleEventPublisher.publish(event)
             → IScheduleEventListener.onEvent()  (异常隔离)
排程失败 → ScheduleEvent.failed(context, reason)
          → 同上
结果发布 → ScheduleEvent.published(context)
          → 同上
```

**事件发布位置**：

- `ResultValidationHandler`：排程成功时发布 `SCHEDULE_COMPLETED`
- `AbsLhScheduleTemplate`：异常捕获后发布 `SCHEDULE_FAILED`
- `LhScheduleServiceImpl.publishSchedule()`：手动发布时发布 `RESULT_PUBLISHED`

### 5.3 原子持久化机制：`SchedulePersistenceService`

```java
@Transactional(rollbackFor = Exception.class)
public void replaceScheduleAtomically(LhScheduleContext context) {
    // 1. 二次校验已发布（防并发窗口）
    // 2. DELETE 旧排程结果 (WHERE factoryCode + scheduleDate + isDelete=0)
    // 3. DELETE 旧未排结果
    // 4. DELETE 旧换模计划
    // 5. DELETE 旧过程日志 (按旧批次号)
    // 6. INSERT 新排程结果（批量）
    // 7. INSERT 新未排结果（批量）
    // 8. INSERT 新换模计划（批量）
    // 9. INSERT 新过程日志（批量）
}
```

**数据一致性保证**：

1. **事务边界**：整个"删旧+写新"在一个事务内完成，保证原子性
2. **先删后写**：先清理旧批次数据，再写入新批次数据
3. **二次防重**：事务开始时再次检查 `isRelease=1`，防止事务外的时间窗口内被发布
4. **按目标日隔离**：删和写都以 `factoryCode + scheduleDate` 为维度，不影响其他日期

### 5.4 配置快照机制：`LhScheduleConfigResolver` + `LhScheduleConfig`

```
LhScheduleConfigResolver.resolveAndAttach(context)
  ├── loadLhParams(factoryCode)        // 从数据库 LhParams 表按工厂加载原始参数
  ├── buildConfig(lhParamsMap)         // 按 "数据库值 > 常量默认值" 解析
  │     ├── putIntValue()              // 整数参数（含最小值保护）
  │     ├── putDoubleValue()           // 浮点参数
  │     └── putStringValue()           // 字符串参数
  └── context.setScheduleConfig(config) // 挂载不可变配置快照
```

**设计要点**：

- 一次排程内配置不变（快照机制），避免运行期间参数表被修改导致排程逻辑口径不一致
- 约 50 个强类型快捷 getter（如 `isForceRescheduleEnabled()`、`isSpecifyMachineRuleEnabled()` 等）
- `getResolvedParamMap()` 返回 `unmodifiableMap`，防止运行时修改

## 七、架构亮点总结

| 维度 | 设计选择 | 效果 |
|------|----------|------|
| **主流程编排** | 模板方法模式 + `final` 方法 | 六步顺序不可变，子类只填充业务逻辑 |
| **算法可扩展** | 11 个策略接口 | 机台匹配、SKU 排序、首检均衡等可独立替换 |
| **数据共享** | 单一 Context 数据总线 | 避免参数层层传递，所有 Handler/Strategy 读写同一实例 |
| **并发安全** | Redis 分布式锁 + Lua 释放脚本 | 同厂同日串行执行，安全释放防误删 |
| **数据一致性** | `@Transactional` 原子替换 | 删旧+写新在同一事务，按目标日隔离 |
| **配置隔离** | 配置快照（不可变） | 一次排程规则口径稳定，不受运行时参数变更影响 |
| **事件通知** | 观察者模式 + 异常隔离 | 排程完成/失败/发布事件解耦通知外部系统 |
| **错误处理** | 分层中断 + 聚合校验 | Handler 层单步中断，数据校验层收集全部错误 |
| **可观测性** | 步骤快照日志 + 日计划台账 | 每个步骤输出关键指标，便于排查"为什么只到某一步" |
