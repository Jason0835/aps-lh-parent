## 二、短期优化详细方案（1-2周）

### 2.1 工单号生成器线程安全修复

**问题分析**：
```java
// 原代码 - 非线程安全
private static int orderSeq = 0;
private String generateOrderNo(LhScheduleContext context) {
    int seq = (++orderSeq) % 1000;  // 多线程下可能重复
    return String.format("%s%s%03d", "LHGD", dateStr, seq);
}
```

**解决方案**：
- 创建独立的 `OrderNoGenerator` 组件
- 支持 Redis 原子递增（推荐分布式环境）
- 本地 AtomicInteger 降级方案

**新增文件**：`/aps-lh/src/main/java/com/zlt/aps/lh/util/OrderNoGenerator.java`

**关键代码**：
```java
@Slf4j
@Component
public class OrderNoGenerator {
    @Resource
    private StringRedisTemplate stringRedisTemplate;
    
    private final AtomicInteger localSequence = new AtomicInteger(0);
    private volatile boolean useRedis = true;
    
    public String generateOrderNo(Date targetDate) {
        String dateStr = new SimpleDateFormat("yyyyMMdd").format(targetDate);
        int seq = getNextSequence("LHGD", dateStr);
        return String.format("%s%s%03d", "LHGD", dateStr, seq % 1000);
    }
    
    private int getNextSequence(String prefix, String dateStr) {
        if (useRedis) {
            try {
                String redisKey = "APS:LH:ORDER:" + prefix + dateStr;
                Long seq = stringRedisTemplate.opsForValue().increment(redisKey);
                stringRedisTemplate.expire(redisKey, 24, TimeUnit.HOURS);
                return seq != null ? seq.intValue() : 1;
            } catch (Exception e) {
                useRedis = false;  // 降级为本地模式
            }
        }
        return localSequence.incrementAndGet();
    }
}
```

**修改文件**：
- `NewSpecProductionStrategy.java` - 注入 OrderNoGenerator
- `ContinuousProductionStrategy.java` - 注入 OrderNoGenerator

### 2.2 统一收尾判定策略

**问题分析**：
收尾判定逻辑分散在两处，条件不一致：
- `ScheduleAdjustHandler.markEndingSkus()` - 基于班次产能判定
- `ContinuousProductionStrategy.scheduleContinuousEnding()` - 基于日产能判定

**解决方案**：
创建统一的收尾判定策略接口和实现。

**新增文件**：
- `/aps-lh/src/main/java/com/zlt/aps/lh/engine/strategy/IEndingJudgmentStrategy.java`
- `/aps-lh/src/main/java/com/zlt/aps/lh/engine/strategy/impl/DefaultEndingJudgmentStrategy.java`

**关键逻辑**：
```java
@Override
public boolean isEnding(LhScheduleContext context, SkuScheduleDTO sku) {
    // 规则1：已明确标记为收尾
    if (SkuTagEnum.ENDING.getCode().equals(sku.getSkuTag())) {
        return true;
    }
    
    // 规则2：硫化余量 <= 排程期内可生产总产能
    int totalScheduleShifts = 8;  // 可配置
    int totalCapacity = sku.getShiftCapacity() * totalScheduleShifts;
    if (sku.getSurplusQty() <= totalCapacity && sku.getSurplusQty() > 0) {
        return true;
    }
    
    // 规则3：待排量 < 日产能（非满产运行）
    int pendingQty = sku.getPendingQty() > 0 ? sku.getPendingQty() : sku.getSurplusQty();
    if (sku.getDailyCapacity() > 0 && pendingQty < sku.getDailyCapacity() && pendingQty > 0) {
        return true;
    }
    
    return false;
}
```

**问题分析**：

**解决方案**：
使用策略自注册机制。

**修改文件**：`ScheduleStrategyFactory.java`

**改进前**：
```java
@PostConstruct
public void init() {
    strategyCache.put(ScheduleTypeEnum.CONTINUOUS.getCode(),
            productionStrategyMap.get("continuousProductionStrategy"));  // 硬编码
    strategyCache.put(ScheduleTypeEnum.NEW_SPEC.getCode(),
            productionStrategyMap.get("newSpecProductionStrategy"));
}
```

**改进后**：
```java
@PostConstruct
public void init() {
    if (productionStrategies != null) {
        for (IProductionStrategy strategy : productionStrategies) {
            String strategyType = strategy.getStrategyType();
            if (strategyType != null && !strategyType.isEmpty()) {
                productionStrategyCache.put(strategyType, strategy);
            }
        }
    }
}
```

**策略接口扩展**：

### 3.3 机台匹配性能优化

**问题分析**：
每个 SKU 都遍历所有机台，时间复杂度 O(n × m)。

**解决方案**：
创建机台索引管理器，支持快速过滤。

**新增文件**：`/aps-lh/src/main/java/com/zlt/aps/lh/engine/index/MachineIndexManager.java`

**索引类型**：
- **英寸范围索引**：TreeMap 快速范围查询
- **状态索引**：按状态快速过滤
- **模台数索引**：按模台数分组

**优化后查询**：
```java
// 多条件组合查询
public List<MachineScheduleDTO> findCandidates(
        BigDecimal minInch, BigDecimal maxInch, 
        String status, int minMouldNum) {
    
    // 先按最严格的条件过滤（状态）
    List<MachineScheduleDTO> candidates = findByStatus(status);
    
    // 再按英寸范围过滤
    if (minInch != null && maxInch != null) {
        candidates.removeIf(m -> 
            minInch.compareTo(m.getDimensionMinimum()) < 0 || 
            maxInch.compareTo(m.getDimensionMaximum()) > 0
        );
    }
    
    // 最后按模台数过滤
    if (minMouldNum > 0) {
        candidates.removeIf(m -> m.getMaxMoldNum() < minMouldNum);
    }
    
    return candidates;
}
```

## 五、风险评估与兼容性

### 5.1 风险评估

| 风险项 | 影响程度 | 缓解措施 |
|--------|----------|----------|
| OrderNoGenerator Redis 故障 | 中 | 本地 AtomicInteger 降级 |
| 策略工厂改动影响现有代码 | 低 | 保持向后兼容的接口 |
| 规则引擎缓存一致性 | 低 | 定时刷新 + 手动清除 |
| 索引构建耗时 | 低 | 增量更新 + 异步构建 |

### 5.2 兼容性保障

**向后兼容策略**：
1. 保留原接口方法，新增 default 实现
2. 新组件可选注入，不影响现有逻辑
3. 规则引擎默认值与原常量一致

**迁移策略**：
1. 新功能通过开关控制启用
2. 灰度发布，逐步切换
3. 保留旧代码路径，便于回滚

### 5.3 测试验证方案

**单元测试**：
```java
@Test
void testOrderNoGenerator_ThreadSafety() {
    OrderNoGenerator generator = new OrderNoGenerator();
    int threadCount = 100;
    Set<String> orderNos = ConcurrentHashMap.newKeySet();
    
    // 多线程并发生成
    CountDownLatch latch = new CountDownLatch(threadCount);
    for (int i = 0; i < threadCount; i++) {
        new Thread(() -> {
            orderNos.add(generator.generateOrderNo(new Date()));
            latch.countDown();
        }).start();
    }
    
    latch.await();
    assertEquals(threadCount, orderNos.size());  // 无重复
}
```

**集成测试**：
1. 执行完整排程流程，验证结果一致性
2. 对比优化前后排程结果
3. 性能基准测试

**回归测试**：
