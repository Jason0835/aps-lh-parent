# 3302002654 正式非收尾满排目标量改造实施方案

## 1. 明确结论

推荐方案：

仅对**新增排产 SKU** 的 **正式/量试 + 非收尾** 场景实施文档口径修正，在新增排产链路内把目标量收敛为“排程窗口内月计划 dayN 累加值”，并保留“达到最低目标量后，已开机台当天剩余班次补满班产”的规则。

不建议直接修改当前项目“满排模式”的公共目标量语义，不建议把该逻辑全局扩散到续作、type block、收尾等链路。

## 2. 改造目标

本次改造要解决两个核心问题：

1. 当前项目在满排模式下，正式/量试非收尾 SKU 的目标量使用的是“理论窗口产能上限”，与文档要求不一致。
2. 对于 `3302002654` 这类 `day1=100, day2=100, day3=100`、班产 `17`、余量充足的新增排产 SKU，期望结果应允许形成：
   - `K2024: C2~C8 = 119`
   - `K1111: C2~C8 = 119`
   - `K1113: C2~C6 = 85`

即：

- 最低目标量先满足 `300`
- 正规/量试非收尾场景下，已开机台当天剩余班次继续补满

## 4. 设计思路

核心思路是“文档口径下沉到新增排产策略，不改公共满排总语义”。

具体设计如下：

1. 保留 `ScheduleAdjustHandler + TargetScheduleQtyResolver` 现有公共初始化逻辑，避免影响续作和其他策略。
2. 在 `NewSpecProductionStrategy` 内部，对“正式/量试 + 非收尾 + 新增排产”单独解析一个**新增排产最低目标量**：
   - 优先取 `windowRemainingPlanQty`
   - 无继承/无欠产扰动时等价于 `windowPlanQty`
   - 对本案例即 `100 + 100 + 100 = 300`
3. 多机台拆量时：
   - 扩机判断按该最低目标量推进
   - 非最后机台优先整日满排
   - 尾机台在达到最低目标量后，若仍存在“当天剩余班次”，继续补满当天班产
4. 结果校验同步使用“新增排产最低目标量 + 合法补满边界”校验，避免把文档允许的补满结果判成超排

这样可以同时满足：

- 文档要求
- `3302002654` 期望结果方向
- 最小改动范围

## 5. 详细实施步骤

### 步骤 1：新增排产链路引入“最低目标量”口径

改造点：

- `NewSpecProductionStrategy`

处理方式：

- 新增一个仅供新增排产使用的私有方法，例如：
  - `resolveNewSpecFormalMinimumTargetQty(...)`
- 仅在以下条件命中时启用：
  - 新增排产
  - 正式/量试
  - 非收尾
  - 非严格目标量

取值规则：

- 优先取 `windowRemainingPlanQty`
- 如果没有继承/欠产扰动，可等价理解为 `windowPlanQty`
- 不再取理论窗口产能上限

### 步骤 2：调整新增排产多机台拆量锚点

改造点：

- `resolveDynamicMachinePlanQtyByDailyCapacity(...)`
- `resolveRequiredMachineCountByDailyCapacity(...)`
- `resolveCandidateTargetQty(...)`

处理方式：

- 这些方法目前围绕 `targetQty / remainingTargetQty` 工作
- 将正式/量试非收尾的新增排产目标量锚点切到“最低目标量”
- 保持现有 dayN 扩机推导方式不变，只替换其上游目标量口径

预期效果：

- `3302002654` 不会再被压回 `136`
- 会朝 `300` 的最低目标量推进机台分配

### 步骤 3：保留“已开机台当天补满”能力

改造点：

- `resolveMachinePlanQty(...)`
- 当前尾机台补满当天班次的私有逻辑

- 正规/量试非收尾场景下：
  - 达到最低目标量后
  - 已开机台当天剩余班次继续补满
- 注意严格限制为“当天”，不能扩成“整个窗口全部跑满”

预期效果：

- 第三台如 `K1113` 可在当天补到 `85`
- 不会错误膨胀成跨多日的整窗口满排

## 6. 数据处理说明

### 6.1 初始化

- 月计划窗口 dayN 累加值：
  - 由 `windowPlanQty` 表示
- 扣减继承量、叠加欠产传导后的滚动账本：
  - 由 `dailyPlanQuotaMap`
  - `windowRemainingPlanQty` 表示账本汇总剩余量

### 6.2 目标量

建议区分两个口径：

1. 公共 `targetScheduleQty`
   - 保持现有兼容语义，避免影响其他策略

2. 新增排产最低目标量
   - 新增排产正式/量试非收尾专用
   - 来源：`windowPlanQty / windowRemainingPlanQty`

### 6.3 分配

- 多机台扩机判断按“新增排产最低目标量”推进
- 每台机台排完后继续通过 `dailyPlanQuotaMap` 消费滚动账本
- 当天补满产生的超出最低目标量部分，仍按现有 `shiftFillOverQty` 语义记录

### 6.4 回写

- 结果仍落 `t_lh_schedule_result`
- 日计划消费仍走 `applyBlockToDailyQuota(...)`
- 结果校验按新增口径同步校验

## 7. 边界场景

本次需要重点验证以下场景：

1. 正规非收尾，dayN 累加目标量明显大于当前单机窗口量
2. 正规非收尾，达到最低目标量后尾机台当天仍有剩余班次
3. 量试非收尾，逻辑应与正规一致
4. 试制非收尾，不允许因为本次改造被放开超排
5. 收尾场景，不允许被误套用“dayN 累加目标量”
6. 续作场景，不允许因为公共目标量变化被连带放大
7. 存在继承量、欠产传导时，最低目标量与滚动账本是否一致
8. 满排模式下只有 1 台候选机台时，不能错误按 300 强行排到超出真实窗口容量

## 8. 风险点

### 风险 1：误伤续作和其他策略

如果直接改 `TargetScheduleQtyResolver.resolveInitialTargetQty(...)`，会影响：

- 续作
- type block
- 收尾判断
- 结果校验

这是本次最大风险。

### 风险 2：把“当天补满”误放大成“窗口补满”

文档只要求：

- 已开机台**当天剩余班次**补满

不要求：

- 已开机台后续整个窗口全部补满

如果范围放大，会再次出现错误扩产。

### 风险 3：真实窗口产能与示例不完全一致

即使目标量切到 `300`，实际是否精确落成 `119/119/85`，还取决于：

- 换模起点
- 首检占用
- 机台排序
- 工作日历
- 真实窗口班次数

所以验收应以“符合文档语义”优先，而不是只盯死某一组固定机台组合。
