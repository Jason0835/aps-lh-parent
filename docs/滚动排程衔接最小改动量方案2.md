业务背景：
硫化排程是滚动排程模式，每次排程为“排 1 天 + 预排 2 天”。

示例：
第一次排程窗口：2026-04-23 ~ 2026-04-25
第二次排程窗口：2026-04-24 ~ 2026-04-26
每个窗口包含 8 个班次。

第二次排程时，需要正确衔接第一次排程的数据：
- 2026-04-23 已进入历史执行区，不再重排，只读取 MES 实际完成量
- 2026-04-24 ~ 2026-04-25 是两次排程的重叠区，优先继承第一次排程结果
- 2026-04-26 是新增尾部日期，正常新增排程

核心目标：
第二次排程不能完全重排，也不能直接照搬第一次排程。
应采用：
MES 实际状态 + 上一次重叠区可继承排程 + 当前新增尾部排程

一、整体处理原则

1. 第二次排程开始时，机台初始状态必须以 MES 实际状态为准。
   优先级：
   MES 当前机台状态
   > 已锁定/已发布排程
   > 上一次预排结果
   > 系统默认空闲状态

2. 上一次排程中与当前窗口重叠的排程明细，可以作为继承候选。
   但必须校验后才能继承，不能无条件复制。

3. 历史日期不再重排，只根据 MES 实际完成量参与月计划扣减。

4. 月计划 SKU 余量必须重新计算，不能因为上一次预排数据导致重复扣减。

5. 同一个 materialCode 如果在 MES 快照中出现在多台机台，不能因为第一台机台消费了该 SKU，就把该 SKU 从候选池中删除。
   应改为按数量扣减：
   - materialCode 仍保留在候选池
   - 每台机台只扣减自己分配到的数量
   - 当剩余量为 0 时才不可继续分配

6. 换模、换活字块不能直接复用上一次排程的标记。
   第二次排程完成后，需要基于当前 run 的最终机台顺序重新计算。

二、需要重点处理的数据

请重点检查并修改以下相关逻辑：

1. 计划量
   - 第二次窗口开始前的 MES 实际完成量需要扣减月计划
   - 上一次预排但未实际执行的数量不能提前扣减
   - 当前 run 继承成功后，再扣减对应计划量
   - 新增尾部日期正常消耗剩余计划量

2. 月计划 SKU
   - 月计划 SKU 不能按 materialCode 一次性删除
   - 改为维护 remainQty / allocatedQty
   - 多机台续作时，同一 materialCode 可以被多台机台分配，只要 remainQty > 0

3. 机台
   - 当前窗口第一条任务的前置状态来自 MES
   - 包括当前在机物料、当前模具、当前活字块、剩余量、预计收尾时间、设备状态
   - 上一次排程结果只能作为重叠区继承候选

4. 续作
   - 续作优先依据 MES 当前在机 SKU 判断
   - 如果 MES 当前在机 SKU 在月计划中仍有余量，且机台具备生产能力，则优先续作
   - MES 续作优先级高于上一次预排继承
   - 如果 MES 续作与上一次继承计划冲突，以 MES 续作为准，被冲突的继承计划释放回待排池

5. 换模
   - 第二次排程完成后，按每台机台的最终排程顺序重新计算
   - 第一条任务的前置模具来自 MES 当前模具
   - 后续任务的前置模具来自上一条排程任务对应模具
   - 只有当前后 SKU / 模具关系未变化时，才允许继承原换模结果，否则必须重算

6. 换活字块
   - 第二次排程完成后，按每台机台最终排程顺序重新计算
   - 第一条任务的前置活字块来自 MES 当前活字块
   - 后续任务的前置活字块来自上一条排程任务
   - 续作收尾机台选择下一个 SKU 时，优先减少换活字块
   - 优先级建议：
     1）同胎胚描述 + 同主花纹
     2）同规格
     3）同模具
     4）月计划优先级
     5）交期/需求优先级
     6）产能利用率

三、滚动衔接流程

请在现有排程入口前增加一个滚动排程上下文构建逻辑，例如：

buildRollingContext(currentWindow, prevRun)

该逻辑需要完成：

1. 读取当前排程窗口
   例如：2026-04-24 ~ 2026-04-26

2. 查询上一次有效排程
   例如：2026-04-23 ~ 2026-04-25

3. 计算两个窗口的重叠区
   overlapStart = max(prevWindowStart, currentWindowStart)
   overlapEnd = min(prevWindowEnd, currentWindowEnd)

   示例结果：
   2026-04-24 ~ 2026-04-25

4. 读取上一次排程中落在重叠区内的排程明细

5. 读取 MES 当前快照
   包括：
   - 机台当前在机 SKU
   - 当前模具
   - 当前活字块
   - 机台状态
   - 当前剩余量
   - 实际完成量
   - 当前预计收尾时间

6. 初始化当前 run 的月计划余量：
   monthRemainQty =
   月计划总量
   - 当前窗口开始前 MES 实际完成量
   - 已锁定且不允许调整的未来计划量

   注意：
   不要扣减上一次排程中的预排数量。
   上一次重叠区明细只有在当前 run 继承成功后，才扣减月计划余量。

7. 先生成 MES 续作任务

8. 再继承上一次重叠区可用明细

9. 继承失败或被 MES 续作冲突的明细，释放回待排池

10. 对以下内容进行局部重排：
   - 继承失败的任务
   - 被 MES 续作冲突释放的任务
   - 新增尾部日期任务
   - 空白班次或空白产能
   - 换模/换活字块变化导致的容量修正

11. 当前 run 排程完成后，统一重算换模和换活字块

四、继承校验规则

上一次重叠区明细只有满足以下条件时，才能继承到当前 run：

1. 该班次仍落在当前窗口内
2. 机台仍可用，未维修、未停机、未禁用
3. 机台仍具备生产该 SKU 的能力
4. SKU 在当前月计划中仍有剩余量
5. 与 MES 当前机台状态不冲突
6. 与当前 run 中已生成的 MES 续作任务不冲突
7. 继承后不会造成同机台时间重叠
8. 继承后不会造成月计划超量

继承成功时：
- 新建当前 run 的排程明细
- 保留原 machineCode、materialCode、shift、seqNo、planQty
- 写入 sourceRunId / sourceDetailId
- rollingType = INHERIT
- 扣减当前 run 的 monthRemainQty

继承失败时：
- 不复制该明细
- 将 materialCode 和 planQty 释放回待排池
- 后续参与局部重排

五、锁定策略

请按区域设置不同稳定性策略：

1. 当前正式排产日，例如 2026-04-24
   lockLevel = HARD
   原则：稳定优先，除非 MES 冲突，否则尽量不改

2. 预排第 1 天，例如 2026-04-25
   lockLevel = SOFT
   原则：优先继承，允许局部优化

3. 新增尾部日期，例如 2026-04-26
   lockLevel = NONE
   原则：正常新增排程

七、伪代码参考

请参考以下流程实现，不要求完全一致，但业务逻辑必须满足：

public ScheduleResult rollingSchedule(ScheduleRequest request) {

    TimeRange currentWindow = request.getWindow();

    ScheduleRun prevRun =
        scheduleRunRepository.findLatestBefore(currentWindow.getStart());

    MesSnapshot mesSnapshot =
        mesService.getSnapshotBefore(currentWindow.getStart());

    Map<String, Integer> monthRemainQty = buildMonthRemainQty(
        request.getMonthPlans(),
        mesSnapshot.getCompletedQty(),
        getLockedFutureQty(currentWindow)
    );

    TimeRange overlap = calcOverlap(prevRun.getWindow(), currentWindow);

    List<ScheduleDetail> prevOverlapDetails =
        scheduleDetailRepository.findByRunAndTimeRange(
            prevRun.getId(),
            overlap
        );

    RollingContext context = new RollingContext();
    context.setCurrentWindow(currentWindow);
    context.setPrevRun(prevRun);
    context.setOverlap(overlap);
    context.setMesSnapshot(mesSnapshot);
    context.setMonthRemainQty(monthRemainQty);
    context.setPrevOverlapDetails(prevOverlapDetails);

    createMesContinueTasks(context);

    inheritValidOverlapDetails(context);

    buildReleasedDemandPool(context);

    scheduleRemainingDemand(context);

    recalculateChangeMoldAndWordBlock(context);

    return saveCurrentRun(context);
}

九、异常场景必须覆盖

请确保以下场景被正确处理：

1. 上一次预排 SKU 在当前 run 月计划余量不足

6. 当前活字块状态和上一次预排不一致

7. 继承任务与 MES 续作冲突

8. 新增尾部日期
