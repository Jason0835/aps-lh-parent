## 一、代码架构层面分析

### 1.1 整体架构设计评估

#### 架构优点

**1. 分层清晰，职责明确**

```
Controller层 → Service层 → Executor层 → Template层 → Handler层 → Strategy层 → Mapper层
```

- **Controller层** ([LhScheduleController.java](aps-lh/src/main/java/com/zlt/aps/lh/controller/LhScheduleController.java))：简洁的HTTP入口，仅负责请求接收和响应返回
- **Service层** ([LhScheduleServiceImpl.java](aps-lh/src/main/java/com/zlt/aps/lh/service/impl/LhScheduleServiceImpl.java))：构建上下文、委托执行，职责单一
- **引擎层**：通过模板方法+装饰器+策略模式组合实现核心调度

**2. 设计模式应用得当**

| 模式 | 应用位置 | 评价 |
|------|----------|------|
| **模板方法** | [AbsLhScheduleTemplate.java](aps-lh/src/main/java/com/zlt/aps/lh/engine/template/AbsLhScheduleTemplate.java) | ⭐⭐⭐⭐⭐ 优秀。定义六步流程骨架，`final`修饰确保算法结构不变 |
| **策略模式** | 各`I*Strategy`接口 | ⭐⭐⭐⭐ 良好。策略接口定义清晰，支持运行时替换 |
| **装饰器模式** | [ScheduleExecutorConfig.java](aps-lh/src/main/java/com/zlt/aps/lh/engine/decorator/ScheduleExecutorConfig.java) | ⭐⭐⭐⭐⭐ 优秀。非侵入式增强日志、性能监控 |
| **责任链模式** | [DataValidationChain.java](aps-lh/src/main/java/com/zlt/aps/lh/engine/chain/DataValidationChain.java) | ⭐⭐⭐⭐ 良好。支持聚合/短路两种策略，分组校验 |
| **观察者模式** | [ScheduleEventPublisher.java](aps-lh/src/main/java/com/zlt/aps/lh/engine/observer/ScheduleEventPublisher.java) | ⭐⭐⭐⭐ 良好。事件驱动解耦发布与订阅 |
| **工厂模式** | [ScheduleStrategyFactory.java](aps-lh/src/main/java/com/zlt/aps/lh/engine/factory/ScheduleStrategyFactory.java) | ⭐⭐⭐ 一般。简单工厂，扩展性有限 |

**3. 上下文设计优秀**

[LhScheduleContext.java](aps-lh/src/main/java/com/zlt/aps/lh/context/LhScheduleContext.java) 作为数据总线：
- 结构化组织基础数据、中间结果、输出结果
- 支持参数动态获取 (`getParamValue`, `getParamIntValue`)
- 提供中断机制 (`interruptSchedule`)

#### 架构问题与改进建议

**问题1：模块边界不够清晰**

```java
// ScheduleAdjustHandler.java 承担过多职责
private void doHandle(LhScheduleContext context) {
    adjustPreviousSchedule(context);    // 调整逻辑
    gatherSkuByStructure(context);       // 归集逻辑  
    markEndingSkus(context);             // 标注逻辑
    classifyContinuousAndNewSkus(context); // 分类逻辑
}
```

**建议**：将 `ScheduleAdjustHandler` 拆分为 `PreviousScheduleAdjustHandler` + `SkuClassificationHandler`

**问题2：策略工厂扩展性不足**

```java
// 当前：硬编码策略名称
strategyCache.put(ScheduleTypeEnum.CONTINUOUS.getCode(),
        productionStrategyMap.get("continuousProductionStrategy"));
```

**建议**：使用策略自注册机制

```java
public interface IProductionStrategy {
    String getStrategyType();  // 策略自描述类型
}

@PostConstruct
public void init() {
    for (IProductionStrategy strategy : productionStrategyMap.values()) {
        strategyCache.put(strategy.getStrategyType(), strategy);
    }
}
```

**问题3：Handler与Strategy职责重叠**

`ContinuousProductionHandler` 直接调用 `ContinuousProductionStrategy`，两者命名相似但职责边界模糊。

**建议**：明确命名规范
- `Handler`：流程编排
- `Strategy`：算法实现

**问题4：缺少统一异常处理机制**

当前异常处理分散在各Handler中：

```java
// AbsScheduleStepHandler.java
} catch (Exception e) {
    log.error("[{}] 执行异常", getStepName(), e);
    context.interruptSchedule(getStepName() + "执行异常: " + e.getMessage());
}
```

**建议**：引入领域异常体系 + 全局异常处理器

```java
public class ScheduleException extends RuntimeException {
    private final ScheduleStepEnum step;
    private final ErrorCode errorCode;
}

public enum ScheduleErrorCode {
    MES_RELEASED, DATA_INCOMPLETE, NO_MACHINE_AVAILABLE, ...
}
```

### 1.2 耦合度与内聚性分析

**高内聚组件**（✅）：
- `DefaultCapacityCalculateStrategy`：专注产能计算，单一职责
- `DataValidationChain`：专注数据校验，独立完整
- 各个Mapper接口：数据访问边界清晰

**耦合度较高的组件**（⚠️）：
- `LhScheduleContext`：承载过多数据（15+个Map/List），成为"上帝对象"
- `NewSpecProductionStrategy`：依赖4个其他策略接口

**改进建议**：

```java
// 将Context拆分为多个职责单一的对象
public class ScheduleContext {
    private ScheduleParams params;        // 参数配置
    private BaseDataContainer baseData;   // 基础数据
    private ScheduleState state;          // 中间状态
    private ScheduleOutput output;        // 输出结果
}
```

### 1.3 可扩展性与可维护性评估

**✅ 优点**：
1. 策略模式支持算法热替换
2. 校验器可插拔扩展
3. 观察者模式支持事件监听扩展
4. 参数化配置（通过`LhParams`表）

**⚠️ 待改进**：

1. **缺少配置化规则引擎**：换模时间、首检时间等硬编码在常量类
   ```java
   // LhScheduleConstant.java
   public static final int MOULD_CHANGE_PREHEAT_HOURS = 4; // 硬编码
   ```
   **建议**：引入规则配置表

2. **缺少排程版本控制**：无法回溯历史排程配置

3. **缺少单元测试覆盖**：策略类缺少对应的单元测试

## 二、业务逻辑层面分析

### 2.1 核心业务流程分析

**六步流程设计合理性**：

```
S4.1 前置校验 → S4.2 数据初始化 → S4.3 SKU归集 → S4.4 续作排产 → S4.5 新增排产 → S4.6 结果保存
```

**流程优势**：
- 续作优先：优先保障在产SKU连续性，减少换模
- 清晰分离：续作与新增分离处理，逻辑清晰
- 可中断设计：每步后检查中断标志

**潜在问题**：

**问题1：欠产调整逻辑边界模糊**

```java
// ScheduleAdjustHandler.java:89-97
int deficit = nightPlanQty - nightFinishQty;
if (deficit > 0) {
    int currentClass1Plan = result.getClass1PlanQty() != null ? result.getClass1PlanQty() : 0;
    result.setClass1PlanQty(currentClassPlan + deficit);
}
```

**问题**：未考虑超产情况如何处理，且只处理夜班欠产，早/中班欠产被忽略

**建议**：完善超产冲抵逻辑，明确欠产调整规则

**问题2：收尾判定条件不一致**

代码中有两处收尾判定：

```java
// ScheduleAdjustHandler.java:295 - 基于班次产能
if (sku.getSurplusQty() <= totalCapacity && sku.getSurplusQty() > 0) {
    sku.setSkuTag(SkuTagEnum.ENDING.getCode());
}

// ContinuousProductionStrategy.java:91 - 基于日产能
boolean isEnding = SkuTagEnum.ENDING.getCode().equals(sku.getSkuTag())
        || (dailyCapacity > 0 && pendingQty < dailyCapacity);
```

**建议**：统一收尾判定逻辑，抽取为独立策略

**问题3：胎胚库存调整时机**

```java
// NewSpecProductionStrategy.java:65-76
// 胎胚库存调整在排产之后执行，可能导致已排产计划无法执行
```

**建议**：胎胚库存检查前置，在机台匹配阶段就考虑库存约束

### 2.2 业务规则实现问题

**问题1：换模均衡规则硬编码**

```java
// DefaultMouldChangeBalanceStrategy.java
// 早班8台、中班7台、夜班0台 - 硬编码在常量类
```

**风险**：无法根据生产实际情况动态调整

**问题2：首检均衡时间窗口判断**

```java
// DefaultFirstInspectionBalanceStrategy.java:97-98
return hour < noMouldChangeStartHour
        || (hour == noMouldChangeStartHour - 1 && minute == 0);
```

**问题**：`minute == 0` 条件过于严格，导致19:01-19:59都无法安排首检

**问题3：工单号生成线程安全**

```java
// NewSpecProductionStrategy.java:352-358
private static int orderSeq = 0;

private String generateOrderNo(LhScheduleContext context) {
    int seq = (++orderSeq) % 1000;  // 非线程安全
    return String.format("%s%s%03d", "LHGD", dateStr, seq);
}
```

**风险**：多线程环境下可能生成重复工单号

**建议**：使用 `AtomicInteger` 或 Redis 原子递增

### 2.3 业务场景匹配度评估

**✅ 匹配良好的场景**：
1. T日模型（目标日排程窗口）
2. 续作优先策略（减少换模）
3. 机台寸口/模数约束
4. 共用模保护机制
5. 定点机台限制

**⚠️ 需要完善的场景**：
1. **急单插单**：现有流程不支持中途插单
2. **设备故障响应**：缺少实时状态感知
3. **质量异常处理**：缺少质量事件驱动的排程调整
4. **多工厂协同**：当前仅支持单工厂排程

## 三、排程算法层面分析

### 3.1 算法设计思路分析

**整体算法策略：贪心算法 + 规则优先级**

```
SKU排序 → 机台匹配 → 约束校验 → 班次分配
```

**SKU优先级算法**：

```java
// DefaultSkuPriorityStrategy.java:71-80
Comparator<SkuScheduleDTO> comparator = Comparator
    .comparingInt((SkuScheduleDTO s) -> s.isDeliveryLocked() ? 0 : 1)  // 发货锁定
    .thenComparingInt((SkuScheduleDTO s) -> -s.getDelayDays())          // 延误天数
    .thenComparingInt((SkuScheduleDTO s) -> isEnding(s) ? 0 : 1)        // 收尾标记
    .thenComparingInt((SkuScheduleDTO s) -> -s.getEndingDaysRemaining()) // 收尾日
    .thenComparingInt(this::getSupplyChainPriorityOrder);               // 供应链优先级
```

**评价**：优先级规则设计合理，但固定排序无法适应复杂场景

### 3.2 机台匹配算法分析

```java
// DefaultMachineMatchStrategy.java:225-253
candidates.sort(
    Comparator.comparingInt((MachineScheduleDTO m) -> 
        sku.getSpecCode().equals(m.getPreviousSpecCode()) ? 0 : 1)  // 同规格优先
    .thenComparing((m1, m2) -> {...})  // 收尾时间升序
    .thenComparingInt(m -> isSameInch(...) ? 0 : 1)  // 相同英寸
    .thenComparingDouble(m -> calcInchDistance(...))  // 英寸差距
    .thenComparingInt(MachineScheduleDTO::getCapsuleUsageCount)  // 胶囊使用次数
);
```

**优点**：多维度综合排序，考虑规格、时间、英寸、胶囊等因素

**问题**：
1. **缺少全局最优考虑**：贪心选择可能不是全局最优
2. **权重不可配置**：各维度权重固定，无法调整

**改进建议**：引入加权评分机制

```java
public class MachineScoreCalculator {
    public double calculateScore(MachineScheduleDTO machine, SkuScheduleDTO sku) {
        double score = 0;
        score += specMatchWeight * calcSpecMatchScore(machine, sku);
        score += timeWeight * calcTimeScore(machine, sku);
        score += inchWeight * calcInchScore(machine, sku);
        score += capsuleWeight * calcCapsuleScore(machine, sku);
        return score;
    }
}
```

### 3.3 产能计算与班次分配算法

**产能计算公式**：

```java
// DefaultCapacityCalculateStrategy.java:33-35
int shiftSeconds = LhScheduleConstant.SHIFT_DURATION_HOURS * 3600;
return (shiftSeconds / lhTimeSeconds) * mouldQty;
```

**班次分配算法**：

```java
// NewSpecProductionStrategy.java:229-267 - 贪心分配
for (ShiftInfo shift : shifts) {
    int shiftMaxQty = calculateShiftCapacity(...);
    int shiftQty = Math.min(remaining, shiftMaxQty);
    setShiftPlanQty(result, shift.getShiftIndex(), shiftQty);
    remaining -= shiftQty;
}
```

**问题**：
1. **忽略班次效率差异**：假设每班次效率相同，实际夜班效率可能较低
2. **忽略设备停机时间**：未精确计算班次内的实际可用时间
3. **分配顺序固定**：从第一个班次开始顺序填充，可能导致资源不均衡

**改进建议**：

```java
// 引入班次效率因子
public class ShiftEfficiency {
    private double morningEfficiency = 1.0;
    private double afternoonEfficiency = 0.95;
    private double nightEfficiency = 0.9;
}

// 考虑设备停机的有效产能计算
public int calculateEffectiveCapacity(ShiftInfo shift, List<DevicePlanShut> shutdowns) {
    long availableMs = shift.getDuration();
    for (DevicePlanShut shutdown : shutdowns) {
        availableMs -= calculateOverlap(shutdown, shift);
    }
    return (int)(availableMs / 1000 / lhTimeSeconds * mouldQty * efficiencyFactor);
}
```

### 3.4 算法效率分析

**时间复杂度分析**：

| 阶段 | 操作 | 复杂度 |
|------|------|--------|
| SKU排序 | `Collections.sort()` | O(n log n) |
| 机台匹配 | 遍历机台 + 排序 | O(m log m) 每SKU |
| 班次分配 | 遍历班次 | O(8) = O(1) |
| **总体** | n个SKU × m个机台 | O(n × m log m) |

**性能瓶颈识别**：
1. `ScheduleAdjustHandler.gatherSkuByStructure()`：嵌套循环遍历月计划
2. `DefaultMachineMatchStrategy.matchMachines()`：每个SKU遍历所有机台

**优化建议**：

```java
// 使用索引优化机台匹配
public class MachineIndex {
    private Map<String, List<MachineScheduleDTO>> machineByInchRange;
    private Map<String, List<MachineScheduleDTO>> machineByMouldSet;
    
    public List<MachineScheduleDTO> findCandidateMachines(SkuScheduleDTO sku) {
        // 基于索引快速过滤
        List<MachineScheduleDTO> byInch = machineByInchRange.get(sku.getInchRange());
        List<MachineScheduleDTO> byMould = machineByMouldSet.get(sku.getMouldCodes());
        return intersection(byInch, byMould);
    }
}
```

> **与现网对齐**：工程内对应实现为 `com.zlt.aps.lh.component.MachineIndexManager`（`aps-lh/src/main/java/com/zlt/aps/lh/component/MachineIndexManager.java`）。上列 `MachineIndex` 为示意命名。

## 五、总结

### 整体评价

| 维度 | 评分 | 说明 |
|------|------|------|
| 架构设计 | ⭐⭐⭐⭐ | 分层清晰，模式运用得当，但存在职责边界模糊问题 |
| 业务实现 | ⭐⭐⭐⭐ | 核心流程完整，但部分业务规则存在不一致和硬编码 |
| 算法设计 | ⭐⭐⭐ | 贪心策略有效但缺乏全局优化，性能有提升空间 |
| 可扩展性 | ⭐⭐⭐⭐ | 策略模式支持扩展，但配置化和规则引擎不足 |
| 可维护性 | ⭐⭐⭐ | 代码结构良好，但缺少测试覆盖和文档 |

该项目作为一个工业级硫化排程系统，整体架构设计合理，核心业务流程完整。主要改进方向是**细化模块边界**、**提升算法效率**、**增强配置灵活性**。
