# 硫化排程 SKU 多机台 + 日计划控制补充建议方案

> 用途：本文件作为《sku支持多机台方案-DS.md》的补充说明，供 在现有 DS 方案基础上继续实现。  
> 核心补充：在支持“一个 SKU 多机台排产”的同时，保持硫化主流程“一个 SKU 连续排 8 个班次”的排产方式，并增加“按日计划量控制 + 班次剩余产能排满”的约束。

---

## 1. 结论

DS 方案的主方向是可行的，也基本适合硫化排程项目：

1. 采用最小改动方式，不重写主流程，不新增大而全算法框架；
2. 通过多机台合计产能判断收尾，能解决“单台排不完、多台能排完”的场景；
3. 新增 SKU 排产时不再一台机台成功后立即 `break`，而是继续尝试下一台机台，能满足“一个 SKU 支持多机台排产”的基础需求；
4. 非收尾不因胎胚库存上调待排量，收尾才允许上调到胎胚库存，这个方向是正确的。

但是，按照本次补充需求，DS 方案还需要继续增强。当前 DS 方案主要是“排程窗口总量控制”，例如把 `2026-05-07 ~ 2026-05-09` 的 `day7 + day8 + day9` 汇总成窗口目标量后进行多机台拆量。这个口径只能保证窗口总量大致正确，不能保证每一天都尽量按照月计划日计划量排产。

因此，建议在 DS 方案基础上补充一层“SKU + 日期”的日计划额度账本，用它控制每天的计划量消耗；但主排程流程仍然按照“SKU 连续 8 个班次”生成排产块，不改成按天拆开的独立排产任务。

---

## 2. 对 DS 方案的适配性分析

### 2.1 可行点

DS 方案中以下设计建议保留：

| DS 方案点 | 是否保留 | 原因 |
|---|---:|---|
| 多机台合计产能参与收尾判断 | 保留 | 收尾判断不能只看单台机台，需要看 SKU 在窗口内所有可用机台的合计能力。 |
| `NewSpecProductionStrategy.scheduleNewSpecs()` 成功后不立即 `break` | 保留 | 这是支持一个 SKU 拆到多台机台的关键。 |
| 通过 `remainingQty` 控制多机台拆量总量 | 保留但增强 | 单纯总剩余量不够，需要扩展成“总剩余量 + 每日剩余额度”。 |
| 非收尾不使用胎胚库存上调待排量 | 保留 | 避免胎胚库存过大导致普通 SKU 被异常放大排产。 |
| 收尾才允许按胎胚库存上调 | 保留 | 符合硫化收尾逻辑，保证停产/收尾时胎胚尽量清完。 |
| 不重写主流程、不新增大框架 | 保留 | 更符合当前项目最小改动目标。 |

### 2.2 需要补强点

DS 方案需要补强以下几点，否则会和本次补充需求存在偏差：

| 问题 | 说明 | 建议 |
|---|---|---|
| 只控制窗口总量，不控制日计划量 | 例如 day7=100、day8=200、day9=0，窗口总量=300。如果只按 300 排，可能把大量计划提前排到 day7 或 day8。 | 增加 `SKU + 日期` 的日计划额度账本。 |
| `remainingQty` 只有总量，无法知道是哪一天的缺口 | 多机台拆量时，需要知道当前机台产出的班次属于哪一天，并扣减对应日期额度。 | `remainingQty` 保留，同时新增 `dailyRemainingQtyMap`。 |
| 班次排满会导致日计划超出 | 如果当天剩余计划量为 20，但当前班次产能为 40，按需求需要排满 40，则当天会超 20。 | 增加“满班补齐超排量”字段，并优先冲抵窗口内后续日期计划。 |
| 多机台排产结果需要按天汇总校验 | 同一 SKU 可能在多台机台同时排，最终必须按 `materialCode + productionDate` 汇总验证。 | 增加结果校验与日志。 |
| 连续 8 班不能被误解为按天拆任务 | 本项目主流程仍要求一个 SKU 连续排 8 个班次。 | 日计划只做额度控制，不改变连续排产块生成方式。 |

## 3. 核心设计原则

### 3.1 主流程不变：仍按 SKU 连续 8 个班次排产

本次改造不能把 SKU 拆成 `day7`、`day8`、`day9` 三个独立任务分别排产。

正确做法是：

```text
排产策略层：仍然选择 SKU，并在某台机台上生成连续 8 个班次的排产块。
数量控制层：根据排产块中每个班次所属的生产日期，扣减对应日期的日计划额度。
多机台扩展：如果一台机台无法满足该 SKU 在窗口内的日计划需求，则继续选择下一台机台生成另一个连续 8 班排产块。
```

即：

```text
不要：SKU 按 day7/day8/day9 拆成 3 个日任务分别排。
应该：SKU 仍按连续 8 班排；排产结果落到哪一天，就扣哪一天的日计划额度。
```

### 3.2 日计划控制是“账本控制”，不是“排产主粒度”

新增日计划额度账本：

```text
key = materialCode + productionDate
value = 当日计划量、已排量、剩余额度、满班超排量
```

该账本用于控制数量，不改变现有硫化机台匹配、续作、换模、换活字块、首检、保养、清洗等主流程。

### 3.3 一个 SKU 多机台排产时，共用同一个日计划账本

同一 SKU 在多台机台上排产时，不能每台机台各自认为当天还有完整的日计划量。所有机台必须共享同一个 `SKU + 日期` 日计划额度账本。

例如：

```text
SKU-A，2026-05-07 日计划 = 100
机台 M1 已排 60
机台 M2 再排时，2026-05-07 剩余额度只能是 40，而不是 100
```

### 3.4 满班补齐是允许超日计划的例外，但必须记录

业务要求：如果排上的机台班次产能还有，则需要排满这个班次。

因此，当某天剩余日计划量小于当前班次剩余产能时，允许当前班次排满，但超出部分必须记录为“满班补齐超排量”。

示例：

```text
SKU-A，2026-05-07 日计划剩余 = 20
当前班次可排产能 = 40
如果该 SKU 已经排到该班次，则本班次排 40
其中：
- 20 计入 2026-05-07 日计划完成量
- 20 计入满班补齐超排量
```

建议满班补齐超排量优先按以下顺序处理：

1. 优先冲抵排程窗口内后续日期的同 SKU 计划量；
2. 如果窗口内后续日期没有计划量，再冲抵月计划剩余量；
3. 如果月计划剩余量也不足，但业务仍要求排满，则记录为 `shiftFillOverQty`，并打上超计划标识，供后续报表、校验、人工确认使用。

## 4. 建议新增/调整的数据结构

### 4.1 新增 `SkuDailyPlanQuotaDTO`

建议新增 DTO，用于维护某个 SKU 在某一天的计划额度。

```java
public class SkuDailyPlanQuotaDTO implements Serializable {

    private String materialCode;

    /** 生产日期，例如 2026-05-07 */
    private LocalDate productionDate;

    /** 月计划中 day7/day8/day9 对应的日计划量 */
    private int dayPlanQty;

    /** 滚动衔接、已锁定结果、已排结果中已经占用的数量 */
    private int scheduledQty;

    /** 当前日期剩余应排计划量，max(0, dayPlanQty - scheduledQty) */
    private int remainingQty;

    /** 因满班补齐导致的超出当日计划量 */
    private int shiftFillOverQty;

    /** 是否已经完成当日计划 */
    private boolean completed;
}
```

### 4.2 调整 `SkuScheduleDTO`

建议在现有 `SkuScheduleDTO` 中增加每日额度信息。

```java
public class SkuScheduleDTO implements Serializable {

    /** 窗口内每日计划额度，key = productionDate */
    private Map<LocalDate, SkuDailyPlanQuotaDTO> dailyPlanQuotaMap;

    /** 窗口内日计划剩余量汇总 */
    private int windowRemainingPlanQty;

    /** 多机台拆量剩余量，仍可保留，用于快速判断总量是否排完 */
    private int remainingScheduleQty;

    /** 满班补齐导致的窗口内超排量 */
    private int shiftFillOverQty;
}
```

> 注意：`remainingScheduleQty` 不能替代 `dailyPlanQuotaMap`。  
> `remainingScheduleQty` 解决“还剩多少总量没排”，`dailyPlanQuotaMap` 解决“哪一天还剩多少没排”。

### 4.3 调整 `LhScheduleResult`

如果当前结果表已有生产日期、计划量、实际排产量等字段，可以复用；如果没有，建议补充以下字段或至少在内存 DTO 中维护：

```java
/** 该排产结果归属的生产日期，按班次日期计算 */
private LocalDate productionDate;

/** 本结果中属于日计划内的数量 */
private int dailyPlanConsumeQty;

/** 本结果中因满班补齐产生的超日计划数量 */
private int shiftFillOverQty;

/** 是否为满班补齐导致的超排 */
private boolean shiftFillOverFlag;
```

如果不方便改结果表，至少需要在排程上下文中维护中间统计，最终校验与日志可以基于中间统计输出。

## 5. 数量计算规则

### 5.1 初始化日计划额度

以排程日期 `2026-05-09`、排产窗口 `2026-05-07 ~ 2026-05-09` 为例：

```text
SKU-A:
2026-05-07 -> 月计划 day7
2026-05-08 -> 月计划 day8
2026-05-09 -> 月计划 day9
```

初始化公式：

```text
dayPlanQty(date) = 月计划中对应 dayN 的计划量
lockedQty(date) = 滚动衔接已继承结果 + 已锁定结果 + 当轮已排结果
remainingQty(date) = max(0, dayPlanQty(date) - lockedQty(date))
windowRemainingPlanQty = sum(remainingQty(date))
```

### 5.2 非收尾 SKU 目标量

非收尾场景下，目标量应以窗口内每日计划剩余额度为基础，不允许因为胎胚库存大而上调。

```text
nonEndingTargetQty = min(windowRemainingPlanQty, monthRemainQty)
```

其中：

```text
windowRemainingPlanQty = sum(day7/day8/day9 的剩余计划量)
monthRemainQty = 月计划总量 - 已完成量 - 已锁定量 - 当轮已排量
```

### 5.3 收尾 SKU 目标量

收尾场景下，才允许考虑胎胚库存上调。

建议公式：

```text
endingBaseQty = max(windowRemainingPlanQty, embryoStockQty)
endingTargetQty = min(endingBaseQty, monthRemainQty)
```

如果停产/收尾业务明确要求胎胚必须全部清完，并允许超过月计划，则需要参数化控制：

```text
if allowEndingOverMonthPlan = true:
    endingTargetQty = max(windowRemainingPlanQty, embryoStockQty)
else:
    endingTargetQty = min(max(windowRemainingPlanQty, embryoStockQty), monthRemainQty)
```

建议默认不超过月计划余量；只有在停产清胎胚等明确业务场景下，才允许通过参数开启超月计划。

### 5.4 满班补齐量

当排产块已经选择某个 SKU 和某台机台后，该班次应尽量排满。

```text
shiftActualQty = 当前班次可排产能
planConsumeQty = min(shiftActualQty, 当前日期剩余计划量)
shiftFillOverQty = shiftActualQty - planConsumeQty
```

如果 `shiftFillOverQty > 0`，处理顺序如下：

```text
1. 优先冲抵同 SKU 在窗口内后续日期的 remainingQty；
2. 再冲抵同 SKU 月计划剩余量；
3. 仍无法冲抵的部分，记录为满班补齐超排量。
```

## 6. 多机台排产流程建议

### 6.1 总体流程

```text
1. 初始化 SKU 的 dailyPlanQuotaMap；
2. 计算 windowRemainingPlanQty；
3. 调用收尾判断，判断是否为收尾 SKU；
4. 计算 SKU 的 targetScheduleQty；
5. 按现有逻辑匹配候选机台；
6. 在一台机台上尝试生成连续 8 班排产块；
7. 将排产块中的每个班次按 productionDate 回写到 dailyPlanQuotaMap；
8. 如果 SKU 仍有未完成的日计划量或收尾目标量，则继续选择下一台机台；
9. 所有候选机台尝试完后，剩余未排量计入欠产或留待后续滚动排程；
10. 输出结果校验：按 SKU + 日期聚合检查日计划完成量和满班超排量。
```

### 6.2 排产块级别的数量分配

一个连续 8 班排产块可能跨多个日期，所以不能只把排产量记到块开始日期。

建议按班次归属日期拆分：

```text
Block: SKU-A 在 M1 连续排 8 个班次
班次1 -> 2026-05-07 -> qty=40
班次2 -> 2026-05-07 -> qty=40
班次3 -> 2026-05-08 -> qty=40
...
```

每个班次的数量都写入对应日期的 `SkuDailyPlanQuotaDTO`。

### 6.3 多机台继续排产条件

一台机台排完后，不能简单判断“这台成功了就结束”，而应该判断 SKU 在窗口内的目标是否已经满足。

```java
boolean needMoreMachine(SkuScheduleDTO sku) {
    if (sku.getRemainingScheduleQty() > 0) {
        return true;
    }
    return sku.getDailyPlanQuotaMap().values().stream()
            .anyMatch(day -> day.getRemainingQty() > 0);
}
```

建议只有满足以下条件之一时才继续尝试下一台机台：

1. 窗口内仍有日计划剩余额度未完成；
2. 收尾 SKU 仍有胎胚库存/收尾目标量未完成；
3. 停产场景仍有必须清理的胎胚库存。

### 6.4 候选机台选择建议

DS 方案中候选机台匹配逻辑可保留，但多机台场景下建议排序时增加“日计划缺口匹配度”：

```text
优先级建议：
1. 定点机台/续作衔接优先；
2. 能覆盖当前缺口日期的机台优先；
3. 满班超排量更小的机台优先；
4. 换模/换活字块成本更小的机台优先；
5. 原有机台优先级、收尾时间、规格匹配等规则保持不变。
```

## 7. 需要调整的类和方法

### 7.1 `ScheduleAdjustHandler.buildSkuScheduleDTO()`

补充内容：

1. 构建 `dailyPlanQuotaMap`；
2. 按窗口日期读取 `day7/day8/day9`；
3. 扣减滚动衔接已继承量、已锁定结果量；
4. 计算 `windowRemainingPlanQty`；
5. 非收尾不要在这里用胎胚库存上调。

建议伪代码：

```java
Map<LocalDate, SkuDailyPlanQuotaDTO> quotaMap = buildDailyPlanQuotaMap(context, plan, sku);
sku.setDailyPlanQuotaMap(quotaMap);
sku.setWindowRemainingPlanQty(sumRemainingQty(quotaMap));

int basePendingQty = sku.getWindowRemainingPlanQty();
sku.setPendingQty(basePendingQty + carryForwardQty);
```

### 7.2 `TargetScheduleQtyResolver.resolveInitialTargetQty()`

补充内容：

1. 目标量基于 `windowRemainingPlanQty`；
2. 收尾时才考虑胎胚库存上调；
3. 多机台合计产能封顶时，需要考虑“按日计划额度 + 满班补齐”的实际可排量；
4. 返回目标量后，同步设置 `remainingScheduleQty`。

建议伪代码：

```java
int windowRemainingQty = sku.getWindowRemainingPlanQty();
int monthRemainQty = sku.getMonthRemainQty();
int targetQty;

if (sku.isEnding()) {
    int endingBaseQty = Math.max(windowRemainingQty, sku.getEmbryoStockQty());
    targetQty = context.isAllowEndingOverMonthPlan()
            ? endingBaseQty
            : Math.min(endingBaseQty, monthRemainQty);
} else {
    targetQty = Math.min(windowRemainingQty, monthRemainQty);
}

int totalAvailableCapacity = calcSkuTotalAvailableCapacityInWindow(context, sku);
targetQty = Math.min(targetQty, totalAvailableCapacity);
sku.setTargetScheduleQty(targetQty);
sku.setRemainingScheduleQty(targetQty);
```

### 7.3 `DefaultEndingJudgmentStrategy.isEnding()`

DS 方案中“多机台合计产能判断收尾”保留，但需要注意产能计算口径：

```text
不能只算窗口总产能；还要能解释这些产能落在哪些日期、是否能满足窗口内日计划和收尾目标。
```

建议收尾判断仍然先用合计产能做快速判断：

```java
int totalAvailableCapacity = targetScheduleQtyResolver
        .calcSkuTotalAvailableCapacityInWindow(context, sku);

return rule2CandidateQty > 0
        && totalAvailableCapacity > 0
        && rule2CandidateQty <= totalAvailableCapacity;
```

但在最终排产时，以 `dailyPlanQuotaMap` 做实际扣减，避免收尾判断通过后把量集中排到错误日期。

### 7.4 `NewSpecProductionStrategy.scheduleNewSpecs()`

DS 方案中“成功后不立即 break”保留，但需要把单一 `remainingQty` 调整为“总剩余 + 每日剩余”双控制。

建议伪代码：

```java
while (sku.getRemainingScheduleQty() > 0 && hasDailyOrEndingDemand(sku)) {
    MachineScheduleDTO machine = selectNextMachine(context, sku, excludedMachineCodes);
    if (machine == null) {
        break;
    }

    ScheduleBlock block = tryBuildContinuousEightShiftBlock(context, sku, machine);
    if (block == null || block.getTotalQty() <= 0) {
        excludedMachineCodes.add(machine.getMachineCode());
        continue;
    }

    applyBlockToDailyQuota(context, sku, block);
    saveScheduleBlockResult(context, sku, machine, block);
    excludedMachineCodes.add(machine.getMachineCode());
}
```

### 7.5 新增 `applyBlockToDailyQuota()`

该方法是本次补充方案的关键。

职责：

1. 按班次日期拆分排产块数量；
2. 扣减对应日期的 `remainingQty`；
3. 如果班次需要排满，计算 `shiftFillOverQty`；
4. 超出当日计划的部分优先冲抵后续日期额度；
5. 更新 `remainingScheduleQty`。

建议伪代码：

```java
private void applyBlockToDailyQuota(LhScheduleContext context,
                                    SkuScheduleDTO sku,
                                    ScheduleBlock block) {
    for (ScheduleShift shift : block.getShifts()) {
        LocalDate productionDate = shift.getProductionDate();
        int shiftQty = shift.getAvailableCapacity();

        int consumed = consumeDailyQuota(sku, productionDate, shiftQty);
        int overQty = shiftQty - consumed;

        if (overQty > 0) {
            int offsetFutureQty = consumeFutureDailyQuota(sku, productionDate, overQty);
            overQty -= offsetFutureQty;
        }

        if (overQty > 0) {
            sku.setShiftFillOverQty(sku.getShiftFillOverQty() + overQty);
            shift.setShiftFillOverQty(overQty);
            shift.setShiftFillOverFlag(true);
        }

        sku.setRemainingScheduleQty(Math.max(0,
                sku.getRemainingScheduleQty() - shiftQty));
    }
}
```

> 注意：如果业务要求“满班超排量也要扣减月计划余量”，则 `remainingScheduleQty` 可以按 `shiftQty` 扣。  
> 如果业务要求“只有日计划内数量扣减月计划，超排量单独记账”，则应按 `consumed + offsetFutureQty` 扣，`overQty` 单独记录。建议通过参数控制。

### 7.6 `ResultValidationHandler`

建议新增或增强校验：

1. 按 `materialCode + productionDate` 汇总排产量；
2. 对比月计划 dayN；
3. 输出：日计划量、实际排产量、满班补齐超排量、欠产量；
4. 非满班补齐导致的超排应报错或告警；
5. 多机台同 SKU 的汇总量不能重复扣减。

## 8. 与“连续 8 班”的关系说明

本方案不会破坏“一个 SKU 连续排 8 个班次”的主流程。

需要 明确遵守：

```text
1. 不能把 SKU 拆成 day7、day8、day9 三个独立排产任务；
2. 每次选择机台后，仍按现有逻辑生成连续 8 班排产块；
3. 如果同一个 SKU 需要多台机台，则每台机台上也应尽量保持连续 8 班排产；
4. 日计划额度只用于排产块生成后的数量归属和扣减；
5. 一个排产块跨日期时，按班次日期拆分数量，不按排产块开始日期一次性记账。
```

## 9. 示例说明

### 示例 1：一台机台不够，需要多台机台补齐当天计划

```text
排程窗口：2026-05-07 ~ 2026-05-09
SKU-A 月计划：
day7 = 100
day8 = 120
day9 = 80

机台 M1 在 2026-05-07 可排 60
机台 M2 在 2026-05-07 可排 40
```

预期：

```text
M1 排 SKU-A，2026-05-07 贡献 60
M2 排 SKU-A，2026-05-07 贡献 40
SKU-A 在 2026-05-07 合计 = 100，满足 day7 计划
```

### 示例 2：满班补齐导致当天超出日计划

```text
SKU-A，2026-05-07 剩余日计划 = 20
当前机台班次剩余产能 = 40
```

预期：

```text
该班次排满 40
其中：
20 计入 2026-05-07 日计划完成
20 计入满班补齐超排量
如果 2026-05-08 还有 SKU-A 计划缺口，则优先冲抵 2026-05-08 计划
```

### 示例 3：窗口总量正确但日计划错误的反例

```text
SKU-A 月计划：
day7 = 100
day8 = 0
day9 = 200
窗口总量 = 300
```

如果只按窗口总量 300 控制，算法可能在 day7/day8 排满 300，导致：

```text
2026-05-07 超排
2026-05-08 无计划却排产
2026-05-09 欠产
```

因此必须增加 `dailyPlanQuotaMap`，按日期扣减。

## 10. 边界场景验证

| 场景 | 输入 | 预期 |
|---|---|---|
| 单台可排完 | day7=100，M1 可排 100 | 只排 M1，不继续占用其他机台。 |
| 单台不够，多台补齐 | day7=100，M1=60，M2=40 | M1+M2 合计完成 day7=100。 |
| 满班补齐超日计划 | day7 剩余 20，当前班次产能 40 | 排 40，其中 20 记录为满班补齐超排。 |
| 窗口总量不能替代日计划 | day7=100，day8=0，day9=200 | 不允许只按 300 随机落日期，必须按班次日期扣账。 |
| 非收尾胎胚库存大 | day7=100，胎胚库存=500，非收尾 | 目标量不因胎胚库存上调。 |
| 收尾胎胚库存大 | day7=100，胎胚库存=180，月余量=180，收尾 | 目标量允许上调到 180。 |
| 多机台收尾 | 月余量=180，单台产能=100，两台合计=200 | 可判定为收尾，并拆到多台。 |
| 多机台仍排不完 | 目标=300，多台合计=220 | 排 220，剩余 80 进入欠产/后续滚动。 |
| 滚动衔接已有排产 | day7=100，已继承锁定=70 | 当轮 day7 剩余额度只能是 30。 |
| 同 SKU 同日期多机台结果 | M1=60，M2=50，day7=100 | 汇总实际=110，其中 10 必须有满班补齐标识，否则告警。 |

## 11. 风险与处理建议

| 风险 | 说明 | 建议 |
|---|---|---|
| 日计划严格等于计划量与满班排产存在冲突 | 如果班次产能不能刚好凑齐日计划，则严格等于不可达。 | 定义为“按日计划控制，允许满班补齐例外”，并记录超排量。 |
| 超排量是否扣月计划存在争议 | 满班补齐可能导致超过日计划甚至月计划。 | 增加参数控制：是否允许满班超月计划、超出部分是否扣月计划。 |
| 多机台并发更新同一 SKU 日账本 | 同一 SKU 多机台排产时可能重复消费同一天额度。 | 所有机台共享同一个 `dailyPlanQuotaMap`，排产后立即回写。 |
| 结果日期归属错误 | 一个连续 8 班块可能跨日期，不能只记开始日期。 | 按班次生产日期拆分结果数量。 |
| 影响滚动排程衔接 | 第二次滚动排程需要继承第一次已排的日计划消耗。 | 初始化日账本时扣减继承结果和锁定结果。 |
| 影响收尾判断 | 收尾判断通过后，如果日期分布不合理，仍可能日计划偏差。 | 收尾判断看总产能，实际排产扣账看每日额度。 |

## 13. 给 Codex 的补充开发提示词

请基于现有《sku支持多机台方案-DS.md》继续实现以下补充要求，要求最小改动，不重写硫化排程主流程。

### 背景

目前硫化排程主流程是：一个 SKU 一次性连续排 8 个班次。该主流程必须保持，不允许改成按天拆开的独立排产任务。

排程示例：

```text
排程日期：2026-05-09
排产窗口：2026-05-07 ~ 2026-05-09
```

每个 SKU 在排程窗口内，每天排的量需要尽量按照月计划对应的 dayN 计划量执行：

```text
2026-05-07 使用月计划 day7
2026-05-08 使用月计划 day8
2026-05-09 使用月计划 day9
```

例如某 SKU 的 `day7 = 100`，则 `2026-05-07` 该 SKU 的目标排产量应为 100。如果一台机台在当天排不完，需要支持同一个 SKU 在多台机台上排产，共同完成当天计划量。

同时存在一个例外：如果该 SKU 已经排上某台机台某个班次，并且该班次还有剩余产能，则需要排满该班次。由此导致超出当日计划量的部分，需要记录为“满班补齐超排量”，并优先冲抵窗口内后续日期的同 SKU 计划量。

### 实现要求

1. 保留 DS 方案中“支持一个 SKU 多机台排产”的主设计，包括：
   - 收尾判断使用多机台合计产能；
   - `NewSpecProductionStrategy.scheduleNewSpecs()` 成功后不立即 `break`；
   - 使用剩余量控制多机台拆量；
   - 非收尾不因胎胚库存上调；
   - 收尾才允许考虑胎胚库存上调。

2. 在 `SkuScheduleDTO` 增加每日计划额度账本，例如：
   - `Map<LocalDate, SkuDailyPlanQuotaDTO> dailyPlanQuotaMap`；
   - `windowRemainingPlanQty`；
   - `remainingScheduleQty`；
   - `shiftFillOverQty`。

3. 新增 `SkuDailyPlanQuotaDTO`，至少包含：
   - `materialCode`；
   - `productionDate`；
   - `dayPlanQty`；
   - `scheduledQty`；
   - `remainingQty`；
   - `shiftFillOverQty`；
   - `completed`。

4. 在 `ScheduleAdjustHandler.buildSkuScheduleDTO()` 中初始化每日计划额度：
   - 根据排程窗口日期读取月计划 dayN；
   - 扣减滚动衔接继承结果、锁定结果、已排结果；
   - 计算每日 `remainingQty`；
   - 汇总 `windowRemainingPlanQty`；
   - 非收尾不在这里用胎胚库存上调待排量。

5. 调整 `TargetScheduleQtyResolver.resolveInitialTargetQty()`：
   - 非收尾：`targetQty = min(windowRemainingPlanQty, monthRemainQty)`；
   - 收尾：`targetQty = min(max(windowRemainingPlanQty, embryoStockQty), monthRemainQty)`；
   - 如已有停产清胎胚参数允许超月计划，则通过参数控制，不要写死；
   - 最终仍需受多机台合计可用产能限制。

6. 调整 `NewSpecProductionStrategy.scheduleNewSpecs()`：
   - 一台机台排产成功后，不立即移除 SKU；
   - 根据 `remainingScheduleQty` 和 `dailyPlanQuotaMap` 判断是否继续选择下一台机台；
   - 每台机台仍按现有逻辑生成连续 8 班排产块；
   - 一个排产块跨日期时，按班次生产日期拆分数量并扣减对应日期账本。

7. 新增或抽取 `applyBlockToDailyQuota()` 方法：
   - 遍历排产块内每个班次；
   - 根据班次归属日期扣减对应 `SkuDailyPlanQuotaDTO.remainingQty`；
   - 如果当前班次产能大于当日剩余额度，则排满班次，并记录 `shiftFillOverQty`；
   - 超出当日计划的部分优先冲抵窗口内后续日期的同 SKU 计划；
   - 无法冲抵的部分作为满班补齐超排量保留。

8. 增强结果校验：
   - 按 `materialCode + productionDate` 汇总所有机台排产结果；
   - 对比月计划 dayN；
   - 如果实际排产量大于 dayN，必须能解释为满班补齐超排，否则输出告警；
   - 如果实际排产量小于 dayN，计入欠产或后续滚动排程；
   - 同一 SKU 多机台排产时不能重复扣减日计划额度。

9. 增加关键日志：
   - 初始化每日计划额度日志；
   - 每台机台排产前后的 `remainingScheduleQty`；
   - 每个班次扣减的日期、计划内消耗量、满班超排量；
   - 多机台继续排产/停止排产原因；
   - 最终 SKU + 日期汇总校验结果。

### 禁止事项

1. 不要把 SKU 按日期拆成多个独立排产任务；
2. 不要移除现有连续 8 班主流程；
3. 不要仅使用窗口总量 `day7 + day8 + day9` 控制排产，而忽略每天的 dayN 计划；
4. 不要让每台机台独立维护日计划额度，必须同一 SKU 多机台共享同一个日计划账本；
5. 不要让非收尾 SKU 因胎胚库存上调待排产量。

## 14. 最终推荐

建议采用“DS 方案 + 本补充方案”的组合：

```text
DS 方案解决：一个 SKU 如何拆到多台机台上排。
本补充方案解决：拆到多台机台后，每天的计划量如何受控，以及满班补齐如何记录。
```

也就是说，最终落地方案应为：

```text
连续 8 班排产块不变
+ SKU 多机台拆量
+ SKU/日期日计划额度账本
+ 班次排满超排记录
+ SKU/日期维度结果校验
```

该方案既满足硫化排程“连续排产、减少频繁切换”的业务特点，也满足“一个 SKU 支持多机台排产”和“按月计划 dayN 控制每日排产量”的需求。
